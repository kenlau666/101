## Before You Start: What Go Actually Is

Go is a **statically typed, compiled, garbage-collected** programming language designed at Google and released to the public in 2009. Three of its designers — Robert Griesemer, Rob Pike, Ken Thompson — were systems programmers from Bell Labs and elsewhere with decades of opinions about what went wrong with C++ and Java. Go is the language they designed in response.

The defining feature of Go is what it deliberately *doesn't* have. There are no classes. There is no inheritance. There are no generics in the traditional template-style sense (Go did add type parameters in 2022, but the system is much simpler than C++ templates or Rust generics). There are no exceptions. There is no `null` (well, there is `nil`, with caveats). There is no operator overloading. There are no macros. There are no implicit conversions between numeric types. There is no `while` loop — only `for`. There is no ternary operator. The language specification fits in roughly 80 pages.

This is not laziness or oversight. The Go team famously argued for years that adding features adds complexity, and complexity is the enemy of large engineering teams. A junior engineer joining your project can read the entire Go specification in a weekend and be productive Monday. There are essentially no advanced features to discover — what you see is what you get. This is a profound design choice, and it shapes everything about how Go code looks and how Go projects evolve.

What Go *does* have, prominently:

* **Goroutines and channels.** A concurrency model based on Communicating Sequential Processes (CSP), inspired by Tony Hoare's 1978 paper. You can spawn millions of lightweight concurrent threads, and you coordinate them by passing values through channels. This is Go's killer feature, and the reason most people use the language.
* **A garbage collector.** Concurrent, low-pause, well-tuned. Pauses are typically under 1ms. You don't think about memory most of the time.
* **A first-class standard library.** `net/http`, `encoding/json`, `database/sql`, `crypto/...`, `os`, `io` — the standard library is comprehensive and well-designed. You can build production servers with no third-party dependencies at all.
* **Fast compilation.** Even a million-line Go program compiles in seconds. This is one of the reasons Google funded the language: their C++ build times had reached existential levels.
* **Single static binaries.** `go build` produces a single executable with no runtime dependencies. Deploying a Go program is `scp` or a Docker `COPY`. No JVM. No interpreter. No shared libraries.
* **Excellent tooling out of the box.** `go fmt`, `go test`, `go vet`, `go doc`, `go mod` all ship with the compiler. The toolchain is opinionated and unified.

What is Go *for*? Empirically, in 2026: backend services, CLI tools, infrastructure software, devops tooling. The vast majority of containers running Kubernetes are Go. Docker is Go. Terraform is Go. CockroachDB is Go. Most of the Cloud Native Computing Foundation's projects are Go. If you're building an HTTP server, an internal CLI, a daemon, or anything that lives between systems, Go is one of the natural choices, and often the best one.

Go is also explicitly *not* for some things. It's not great for systems programming where you need control over memory layout (use Rust or C++). It's not great for numerical computing or science (use Python or Julia). It's not really used for desktop or mobile apps. Web frontends are still JavaScript-or-WASM territory, not Go.

If you're coming from Java or C#: Go will feel familiar but stripped down. You'll miss generics initially. You'll be surprised how often you don't actually miss them.

If you're coming from Python or JavaScript: Go's static typing will feel rigid at first, then liberating. You'll get used to the compiler catching things your test suite would have caught at 3 AM.

If you're coming from Rust (per the previous course): Go will feel like a vacation. The borrow checker is gone. The garbage collector takes care of memory. Variables are mutable by default. Concurrency is easier — but the safety guarantees are weaker. You'll find yourself relaxing, then occasionally remembering Rust's checks would have caught something Go shipped.

This 101 covers the language, idioms, and concurrency model — enough that you can write production-quality Go after the course. It does *not* cover web frameworks (use the standard library `net/http` plus `chi` or `gin` if you must), database drivers, or cloud SDKs. Those you'll learn when you need them, in an afternoon each, because that's how Go works.

Today's stable Go is **1.26**, released February 2026. Go's release cadence is six months; new versions don't break old code (Go 1's compatibility promise is explicit and unusually well-honored). Anything you learn here will work fine in Go 1.27, 1.28, and beyond.

Plan on 5–8 hours per lesson, including drills. Three lessons. The whole course is roughly a week of evening study, less if you've used a similar language before. Go is genuinely easy to learn — that's its point.

## A Small Glossary You'll See A Lot

* **Package** — Go's unit of code organisation. A directory containing `.go` files that share a `package` declaration. Like a module in Python or a namespace in Java.
* **Module** — a collection of packages versioned together. Roughly a "project." Has a `go.mod` file at its root.
* **`go.mod`** — declares the module's import path and its dependencies. Updated by `go get` and `go mod tidy`.
* **Goroutine** — a lightweight concurrent function execution. You start one with `go funcCall(...)`. Costs ~2KB of stack to start.
* **Channel** — a typed conduit for sending values between goroutines. Created with `make(chan T)`.
* **Slice** — Go's primary collection type. Like a dynamically-resizing array. Most "arrays" you'll write are actually slices.
* **Map** — a hash map. Built into the language with the `map[K]V` syntax.
* **Interface** — a set of method signatures that any type can implement, implicitly. Go's polymorphism mechanism.
* **Struct** — a record type with named fields, like in C.
* **Receiver** — the special parameter that makes a function a method. The thing before the function name in `func (r Receiver) MethodName()`.
* **Zero value** — every type has a default value used when nothing is assigned. `int` is `0`, `string` is `""`, pointers and interfaces are `nil`. This matters more than you'd think.
* **`nil`** — the zero value for pointers, interfaces, slices, maps, channels, and functions. Not the same as null in other languages — see Lesson 1.
* **`error`** — a built-in interface type representing an error condition. The convention is to return errors as the last return value.
* **`any`** — an alias for `interface{}`. The "any type" type. Use sparingly.
* **Defer** — a statement that schedules a function call to run when the surrounding function returns. Used heavily for cleanup.
* **Race** — a data race. Two goroutines accessing the same memory, at least one writing, without synchronisation. Bad. Detectable with `go run -race`.
* **`go fmt`** — the standard formatter. Run it. Always. There is one true Go style.
* **`go vet`** — a static analyser that catches common bugs. Run it on every commit.
* **GOPATH / GOROOT** — environment variables you'll occasionally see. Mostly irrelevant since Go modules; ignore until you're not allowed to.

---

## Key Concepts

* Go keeps the language deliberately small and boring so large teams can read and maintain code without hidden machinery.
* Goroutines and channels are Go's CSP-style concurrency tools for coordinating work by passing values.
* The zero value makes every variable immediately usable, even before you assign anything explicit.
* Go uses composition, structs, methods, and implicit interfaces instead of classes and inheritance.
* Errors are ordinary values returned from functions; there are no exceptions to catch.

# Lesson 1: The Language and Its Shape

## 1.1 Why This Lesson Exists

You can probably learn the syntax of Go in two hours. Most of the constructs are familiar — `if`, `for`, function calls, struct field access. There's nothing genuinely strange in Go's surface syntax until you hit goroutines (Lesson 2). What needs careful explanation is the *shape* of Go programs — how they're organised, what idioms have evolved, what counts as good Go.

Three things in particular trip up newcomers:

1. **The way Go does errors.** No exceptions. Errors are values. They're returned alongside results, checked with `if err != nil`, and propagated by hand. This is repetitive — you'll write `if err != nil { return err }` constantly — and there's no clever way around it. Newcomers often try to fight this; senior Go engineers find the explicitness clarifying.
2. **Interfaces and how they're implemented.** Go interfaces are *implicit* — you don't declare "this type implements that interface." If a type has the right methods, it satisfies the interface. This is unfamiliar to anyone coming from Java or C#, and it has profound implications for how Go libraries are structured.
3. **The lack of features you expect.** No classes. No inheritance. No generics (well, very limited generics since 2022). No exceptions. No `Optional<T>` or `Result<T, E>`. Newcomers spend their first month trying to recreate familiar patterns and being told "no, just use a struct" or "no, just return two values."

This lesson covers types, structs, methods, interfaces, and error handling — the core building blocks of any Go program. The next lesson covers concurrency, which is where Go gets distinctive. The third covers slices, maps, packages, and the standard library — the practical machinery you use every day.

## 1.2 Hello, World, And What's Going On

A Go program looks like this:

```go
package main

import "fmt"

func main() {
    fmt.Println("hello, world")
}
```

Let's walk through every line, because each one carries information.

**`package main`**: every Go file starts with a package declaration. Files in the same directory are part of the same package; you can't mix package names within a directory. The package named `main` is special — it produces an executable. Any other package produces a library.

**`import "fmt"`**: Go's standard library is organised into packages, like `fmt`, `os`, `net/http`, `encoding/json`. You import them by their path. `fmt` provides formatted I/O — Print, Printf, Sprintf, Scanln, etc.

**`func main() { ... }`**: the entry point of an executable program. No arguments, no return value. The function ends; the program exits with status 0.

**`fmt.Println("hello, world")`**: a function call. `Println` is a function in package `fmt`; you reference it as `fmt.Println`. It prints its arguments, separated by spaces, followed by a newline.

To run it: save as `main.go`, then `go run main.go`. Or `go build` to produce a binary you can invoke directly. Or `go install` to put it in `$GOPATH/bin` (or `$HOME/go/bin`).

A few things you might not notice:

* No semicolons. Go's grammar is designed so the lexer inserts them at end of line. You can write them; you shouldn't.
* No parentheses around the `if`/`for` condition: `for i := 0; i < 10; i++ { ... }`, not `for (i = 0; i < 10; i++)`.
* The opening brace must be on the same line as the `func`, `if`, or `for`. This isn't style; it's a hard rule because of the semicolon insertion.
* There are no header files, no forward declarations. The compiler reads all files in a package and figures out what references what.

## 1.3 Variables, Types, and the Zero Value

Go is statically typed. Every variable has a type, known at compile time. The compiler will refuse to compile if types don't match — including refusing to add an `int` to a `int64`, which surprises newcomers.

```go
var x int = 10            // explicit type and initial value
var y int                  // explicit type, zero-valued (y == 0)
var z = 10                 // type inferred from initial value (int)
w := 10                    // shorthand: declare and infer in one step
                           // ":=" only inside functions, not at package level
```

The `:=` form is used constantly; almost every local variable in real Go code uses it. Outside functions, you use `var`.

Multiple assignment works:

```go
a, b := 1, "hello"
x, y = y, x   // swap; no temporary variable needed
```

### Basic types

* `bool` — true / false.
* `int`, `int8`, `int16`, `int32`, `int64` — signed integers. `int` is platform-dependent (64-bit on modern systems).
* `uint`, `uint8`, ..., `uint64` — unsigned integers. `byte` is an alias for `uint8`.
* `float32`, `float64` — floating point. `float64` is the default for `:=` literals like `3.14`.
* `complex64`, `complex128` — complex numbers. Niche.
* `string` — immutable UTF-8 byte sequence. Different from `[]byte`.
* `rune` — alias for `int32`, used to mean "a Unicode code point." A `string` is bytes; iterating with `range` gives you `rune`s.

### The zero value

Every type has a zero value. When you declare a variable without initialising it, it gets the zero value for that type:

```go
var x int        // 0
var s string     // ""
var b bool       // false
var p *int       // nil
var sl []int     // nil — yes, the zero value of a slice is nil
var m map[string]int  // nil
var c chan int   // nil
var i interface{}     // nil
var f func()     // nil
```

This is a *huge* design decision. You're guaranteed that any declared variable is in a usable state. There are no uninitialised variables in Go. This catches an entire category of C/C++ bugs at the language level.

But it has a consequence: nil slices, nil maps, and nil channels behave specially.

* **Nil slices** can be read from (yielding zero items), iterated with `for range` (zero iterations), and `append`ed to (which allocates). You can treat a nil slice like an empty slice for almost all purposes.
* **Nil maps** can be *read* from (returning the zero value of the value type) but **cannot be written to** — that's a runtime panic. This is a famous gotcha. You must `make(map[K]V)` before writing.
* **Nil channels** block forever on send and receive. Sometimes useful intentionally; usually a bug.

We'll come back to all of these. For now: the zero value is your friend, except when it isn't.

### Type conversions

Go has *no implicit conversions* between numeric types. Different from C, very different from JavaScript. You must explicitly convert:

```go
var x int = 10
var y int64 = int64(x)         // explicit conversion required
var z float64 = float64(x)     // also required
// var w int64 = x             // COMPILE ERROR: cannot use x (int) as int64
```

This feels noisy at first. It catches real bugs — accidentally treating a 32-bit integer as 64-bit, accidentally truncating, accidentally mixing units (e.g., bytes vs kilobytes). The Go answer is "be explicit," and over time you stop minding.

### Constants

```go
const Pi = 3.14159
const MaxRetries = 5
const (
    StatusOK    = 200
    StatusError = 500
)
```

Constants are evaluated at compile time. They can be untyped, which makes them more flexible — `Pi` above can be assigned to a `float32` or `float64` depending on context. Once you give them an explicit type (`const X int = 5`), you lose that flexibility.

The `iota` keyword auto-increments constants in a `const` block, used for enum-like patterns:

```go
const (
    Sunday = iota   // 0
    Monday          // 1
    Tuesday         // 2
    Wednesday       // 3
    // ...
)
```

You'll see `iota` constantly in the standard library. It's idiomatic.

## 1.4 Functions

Functions are first-class. They have arguments, return values, and (Go's quirk) can return multiple values.

```go
func add(x, y int) int {
    return x + y
}

func divmod(a, b int) (int, int) {
    return a / b, a % b
}

q, r := divmod(17, 5)
```

When two or more parameters share a type, you can omit the type from all but the last: `func add(x, y int)` not `func add(x int, y int)`. Stylistic; both compile.

### Multiple return values

This is genuinely useful and pervasive. The most common use: returning a result and an error.

```go
func parseInt(s string) (int, error) {
    // ...
}

n, err := parseInt("42")
if err != nil {
    return err
}
// use n
```

Almost every function in the standard library that can fail returns `(T, error)`. We'll dig into errors in section 1.7.

### Named return values

You can name return values. Then `return` without arguments returns them implicitly:

```go
func divide(a, b int) (quotient int, remainder int, err error) {
    if b == 0 {
        err = fmt.Errorf("divide by zero")
        return                 // returns 0, 0, error
    }
    quotient = a / b
    remainder = a % b
    return                     // returns the named values
}
```

Named returns can be useful for documentation (the names appear in `go doc`) and for `defer`-based modification (more on `defer` later). They're also overused by people new to Go, who name everything; experienced Go programmers name returns sparingly, only when it adds clarity.

### Variadic parameters

A function can take a variable number of arguments using `...`:

```go
func sum(nums ...int) int {
    total := 0
    for _, n := range nums {
        total += n
    }
    return total
}

sum(1, 2, 3)       // 6
sum(1, 2, 3, 4, 5) // 15

slice := []int{1, 2, 3}
sum(slice...)      // pass a slice as variadic args with `...`
```

The most famous variadic function is `fmt.Printf("hello %s, age %d", name, age)` — `Printf` takes any number of arguments after the format string.

### Functions are values

You can store functions in variables, pass them as arguments, and return them.

```go
add := func(x, y int) int { return x + y }
result := add(2, 3)

func apply(f func(int) int, x int) int {
    return f(x)
}

double := func(x int) int { return x * 2 }
apply(double, 5)   // 10
```

The function-as-value syntax is verbose compared to languages with arrow functions, but it's clear. Closures work as expected: a function literal can reference variables from the enclosing scope.

```go
func counter() func() int {
    count := 0
    return func() int {
        count++
        return count
    }
}

c := counter()
fmt.Println(c())   // 1
fmt.Println(c())   // 2
fmt.Println(c())   // 3
```

The closure captures `count`. Each call to `counter()` returns a new closure with its own `count`.

## 1.5 Structs and Methods

Structs are how you make compound types. They're like records in Pascal, structs in C, classes-without-methods in Java.

```go
type Order struct {
    ID       int64
    Price    int64
    Quantity int64
    IsBuy    bool
}

o := Order{
    ID:       1,
    Price:    10000,
    Quantity: 5,
    IsBuy:    true,
}

fmt.Println(o.ID)      // 1
o.Price = 12000        // mutate a field
```

Notes:

* Fields starting with a capital letter are *exported* — visible from other packages. Lowercase fields are package-private. This is Go's only access control mechanism. Same rule applies to types, functions, methods, constants.
* Struct literals with named fields (`Order{ID: 1, ...}`) are clearer than positional (`Order{1, 10000, 5, true}`); always use named.
* Trailing commas are required if your struct literal spans multiple lines. `go fmt` enforces this.

You can compare structs with `==` if all fields are comparable:

```go
o1 := Order{1, 100, 5, true}
o2 := Order{1, 100, 5, true}
fmt.Println(o1 == o2)   // true
```

You cannot compare structs containing slices, maps, or functions with `==` — those types aren't comparable, so structs containing them aren't either. For "deep equality" use `reflect.DeepEqual` (slow) or write a method.

### Methods

Methods are functions with a *receiver*. A receiver is the type the method belongs to:

```go
type Order struct {
    ID       int64
    Price    int64
    Quantity int64
}

// Method with value receiver: receives a copy.
func (o Order) Notional() int64 {
    return o.Price * o.Quantity
}

// Method with pointer receiver: receives a pointer; can mutate.
func (o *Order) ReduceQuantity(by int64) {
    o.Quantity -= by
}

func main() {
    o := Order{ID: 1, Price: 100, Quantity: 5}
    fmt.Println(o.Notional())   // 500

    o.ReduceQuantity(2)         // Go auto-takes the address; works on `o`.
    fmt.Println(o.Quantity)     // 3
}
```

Two flavours of receiver, with different semantics:

* **Value receiver `(o Order)`**: the method gets a *copy* of the struct. Mutations don't affect the original. Fine for read-only methods on small structs.
* **Pointer receiver `(o *Order)`**: the method gets a pointer; can mutate the original.

The convention in idiomatic Go: **pick one and stick with it for a given type**. If any method needs a pointer receiver, all methods should use pointer receivers. Mixing is allowed but confusing.

A general rule: use pointer receivers unless the struct is genuinely small (a few primitives) and the method is read-only and called frequently. When in doubt, pointer receiver. The performance overhead of pointer receivers is negligible for nearly all code.

### "There are no classes" — really

Go does not have classes, inheritance, virtual methods, abstract base classes, generics on classes (well, generics in general work differently), or any of the OOP machinery you might be used to.

What it has:

* **Structs with methods** — gives you the "type with behavior" notion.
* **Composition by embedding** — a struct can embed another struct, and the embedded struct's fields and methods become available on the outer struct.
* **Interfaces** — the polymorphism mechanism.

That's it. There is no `class Animal { ... }; class Dog extends Animal { ... }`. There is `type Animal struct{...}` plus interface satisfaction.

This is initially upsetting if you come from Java. Then you realise that 90% of class hierarchies are unnecessary, and what you really wanted was an interface and a function. The Go designers made a deliberate choice that the absence of inheritance pushes you toward better designs. After a year of Go you'll look at a `class FooImpl extends AbstractFooBase` and find it unnecessarily complicated.

### Embedding

A struct can embed another struct anonymously (no field name). The outer struct gains the inner struct's methods and fields:

```go
type Animal struct {
    Name string
}

func (a Animal) Greet() string {
    return "I am " + a.Name
}

type Dog struct {
    Animal      // embedded; no field name
    Breed  string
}

func main() {
    d := Dog{
        Animal: Animal{Name: "Rex"},
        Breed:  "Labrador",
    }
    fmt.Println(d.Name)      // promoted field; works as if Dog had Name directly
    fmt.Println(d.Greet())   // promoted method; works as if Dog had Greet directly
    fmt.Println(d.Breed)     // Dog's own field
}
```

Embedding is *not* inheritance. It's syntactic sugar for "give me access to the embedded thing's stuff." There's no virtual dispatch. Methods on `Animal` don't see `Dog`. If `Dog` defines its own `Greet()`, it shadows Animal's, and there's no `super.Greet()` — you'd write `d.Animal.Greet()` explicitly.

Use embedding for "has-a" relationships where you want delegation. Don't use it to fake inheritance hierarchies.

## 1.6 Interfaces

Interfaces are Go's polymorphism mechanism. They define a set of methods. Any type that has those methods satisfies the interface, *implicitly*.

```go
type Greeter interface {
    Greet() string
}

type English struct{}
func (e English) Greet() string { return "hello" }

type German struct{}
func (g German) Greet() string { return "hallo" }

func sayHi(g Greeter) {
    fmt.Println(g.Greet())
}

func main() {
    sayHi(English{})   // hello
    sayHi(German{})    // hallo
}
```

Note: I never wrote `English implements Greeter`. The compiler checked: does `English` have a `Greet() string` method? Yes. So `English` satisfies `Greeter`. Same for `German`.

This is called **structural typing** or **duck typing at compile time**. It has profound consequences:

* You can make a type satisfy an interface defined in another package, including the standard library, without modifying the type. Want to make your `Order` work with `fmt.Println` formatting? Just give it a `String() string` method, and it satisfies `fmt.Stringer`.
* Interfaces tend to be small. The standard library is full of one-method interfaces (`io.Reader`, `io.Writer`, `fmt.Stringer`, `error`). Go's idiom is "the bigger the interface, the weaker the abstraction" — small focused interfaces, often defined right where they're consumed.
* You can mock anything for testing without specialised mock libraries. Define a small interface for the dependency, swap in a fake implementation.

### The most important interfaces

You'll see these constantly:

```go
// io.Reader: anything you can read bytes from.
type Reader interface {
    Read(p []byte) (n int, err error)
}

// io.Writer: anything you can write bytes to.
type Writer interface {
    Write(p []byte) (n int, err error)
}

// error: anything with an Error() method.
type error interface {
    Error() string
}

// fmt.Stringer: types that have a custom string representation.
type Stringer interface {
    String() string
}
```

`io.Reader` and `io.Writer` are the *single most important abstraction in Go*. Files, network connections, gzip streams, JSON encoders, byte buffers, HTTP request bodies — all of them are Readers and Writers. Functions that operate on Reader can read from any of them. This is what makes Go's I/O ecosystem composable.

```go
import (
    "io"
    "os"
    "strings"
)

func copyToStdout(r io.Reader) error {
    _, err := io.Copy(os.Stdout, r)
    return err
}

func main() {
    // Works for files:
    f, _ := os.Open("input.txt")
    copyToStdout(f)

    // Works for strings:
    copyToStdout(strings.NewReader("hello"))

    // Works for HTTP responses, gzip readers, in-memory buffers, ...
}
```

One function. Many use cases. No inheritance involved.

### The empty interface and `any`

The empty interface `interface{}` has no methods. Every type satisfies it (every type has at least zero methods). It's Go's "anything goes" type.

In modern Go (1.18+), `any` is an alias for `interface{}`. They're identical; `any` reads better.

```go
var x any = 42
var y any = "hello"
var z any = []int{1, 2, 3}
```

You'll meet `any` in places like JSON decoding (where you don't know the type ahead of time) and reflection. Use it sparingly — once you put a value in `any`, you have to use a type assertion or type switch to get it back out, and the compiler can't help you.

```go
var x any = 42
n := x.(int)         // type assertion: panics if x isn't an int
n, ok := x.(int)     // safer: ok is false instead of panicking

switch v := x.(type) {
case int:
    fmt.Println("int:", v)
case string:
    fmt.Println("string:", v)
default:
    fmt.Println("something else")
}
```

The `switch v := x.(type)` is a *type switch* — switch on the dynamic type of an interface value. You'll need it occasionally; don't reach for it casually.

### Interface values, internally

An interface value is *two pointers*: one to the underlying type's information, one to the underlying value. When you call `g.Greet()` on a `Greeter`, the runtime looks up the method via the type pointer and invokes it on the data pointer.

This is dynamic dispatch — slower than a direct function call by a few nanoseconds. Negligible for almost everything. Matters in tight inner loops; you'd switch to concrete types there.

The "two pointers" thing has a surprising consequence: the zero value of an interface is `nil` (both pointers are nil). But an interface value can be "non-nil but holding a nil pointer":

```go
type MyError struct{ msg string }
func (e *MyError) Error() string { return e.msg }

func badFunc() error {
    var e *MyError = nil
    return e            // returning a nil *MyError
}

func main() {
    err := badFunc()
    if err != nil {
        fmt.Println("error:", err)   // PRINTS! err is "non-nil-interface holding nil-pointer"
    }
}
```

This is the famous **typed-nil-interface gotcha**. The interface value `err` has a type (`*MyError`) and a value (`nil`). Type isn't nil, so the interface isn't nil, so `err != nil` is true. But calling `err.Error()` panics because the underlying pointer is nil.

The fix: don't return typed nil pointers as errors. Always return literal `nil`:

```go
func goodFunc() error {
    return nil   // a true nil interface
}
```

Or:

```go
func goodFunc() error {
    var e *MyError = nil
    if e == nil {
        return nil
    }
    return e
}
```

Every Go programmer hits this once. The lesson: interface comparison to `nil` checks both type and value. Be careful when returning interface types from functions that compute them via pointers.

## 1.7 Errors as Values

There are no exceptions in Go. There is no `try` / `catch` / `finally`. There is no `throw`. There is no special control flow for error paths.

Instead: errors are values, returned from functions that can fail, checked with `if`, propagated by hand. This is repetitive on purpose.

```go
func parseAge(s string) (int, error) {
    n, err := strconv.Atoi(s)
    if err != nil {
        return 0, err
    }
    if n < 0 {
        return 0, fmt.Errorf("negative age: %d", n)
    }
    return n, nil
}

func main() {
    age, err := parseAge("25")
    if err != nil {
        fmt.Println("error:", err)
        os.Exit(1)
    }
    fmt.Println("age:", age)
}
```

The `error` type is just an interface:

```go
type error interface {
    Error() string
}
```

Anything with an `Error() string` method satisfies it. The standard library provides `errors.New(...)` and `fmt.Errorf(...)` for creating errors:

```go
errors.New("something went wrong")
fmt.Errorf("invalid order id: %d", id)
```

`fmt.Errorf` supports a special `%w` verb that *wraps* an underlying error:

```go
err := readFile()
if err != nil {
    return fmt.Errorf("loading config: %w", err)
}
```

The `%w` lets callers unwrap to get at the original error with `errors.Is(err, ...)` or `errors.As(err, ...)`:

```go
if errors.Is(err, os.ErrNotExist) {
    // file didn't exist
}

var pathErr *os.PathError
if errors.As(err, &pathErr) {
    // err wraps a *os.PathError; pathErr is filled in
}
```

This `%w` + `errors.Is` + `errors.As` machinery (added in Go 1.13) is the modern idiom for error propagation with context.

### The `if err != nil` rhythm

Real Go code is full of:

```go
result, err := someOp()
if err != nil {
    return nil, err
}
result2, err := otherOp(result)
if err != nil {
    return nil, fmt.Errorf("something failed: %w", err)
}
// continue with result2
```

This is the dominant rhythm of Go. People coming from languages with exceptions find it noisy. People coming from Rust find it strikingly similar to `?`-less Rust (just without the `?` shortcut). After a few months you stop noticing. After a year you find it clarifying — every error path is visible at every step.

Some patterns to know:

**Wrap errors at boundaries.** When an error crosses an abstraction boundary, wrap it with context: "couldn't load config: %w". The chain of wrappings is the stack trace you don't otherwise have.

**Don't log-and-return.** This is a junior mistake: `log.Println(err); return err`. The caller will also log it. You get the same error twice in the logs. Either log *or* return, not both. Usually return.

**Sentinel errors are package-level vars.** `io.EOF`, `os.ErrNotExist`, `sql.ErrNoRows`. Compare with `errors.Is(err, io.EOF)`.

**Custom error types are structs with `Error() string`.** Useful when callers might want structured access to error fields:

```go
type ValidationError struct {
    Field   string
    Reason  string
}

func (v *ValidationError) Error() string {
    return fmt.Sprintf("validation failed for %s: %s", v.Field, v.Reason)
}
```

### Panic and recover (used rarely)

There is one exception-like mechanism: `panic` and `recover`. `panic(value)` aborts the current function and unwinds the stack; `recover()` (called inside a deferred function) intercepts a panic and lets you continue.

```go
func safeCall() {
    defer func() {
        if r := recover(); r != nil {
            fmt.Println("recovered from:", r)
        }
    }()
    panic("something broke")
}
```

When to use panic/recover:

* **In your own code: rarely.** Use `error` for anything a caller might reasonably handle.
* **For genuine bugs, not runtime conditions.** "Index out of bounds" panics; "file not found" returns `error`.
* **At HTTP server boundaries:** the `net/http` server recovers from panics in handlers so a buggy handler doesn't crash the whole server.
* **In testing:** the `testing` package uses panic for test failures.

The rule: errors for things that *can* go wrong; panics for things that *shouldn't*.

## 1.8 Control Flow

Mostly familiar. A few differences worth highlighting.

### `if`

```go
if x > 10 {
    // ...
} else if x > 0 {
    // ...
} else {
    // ...
}
```

Useful Go-specific form: an `if` can declare a variable scoped to the `if` block:

```go
if err := doSomething(); err != nil {
    return err
}
// err not in scope here
```

This is *the* idiom for handling errors. You'll see it constantly. The variable is in scope for both `if` and `else`.

### `for` is the only loop

```go
// C-style for loop
for i := 0; i < 10; i++ {
    fmt.Println(i)
}

// while loop
for x < 100 {
    x *= 2
}

// infinite loop
for {
    if done {
        break
    }
    // ...
}

// for with range — over slices, maps, strings, channels
for i, v := range slice {       // i is index, v is value
    // ...
}
for k, v := range myMap {       // k is key, v is value (random order!)
    // ...
}
for i, c := range "abc" {       // i is byte index, c is rune
    // ...
}
for v := range ch {             // receive from channel until closed
    // ...
}
```

`break` and `continue` work as you'd expect. There's no `do-while`.

### `switch`

```go
switch day {
case "Mon", "Tue", "Wed", "Thu", "Fri":
    fmt.Println("weekday")
case "Sat", "Sun":
    fmt.Println("weekend")
default:
    fmt.Println("unknown")
}
```

Notes:

* No fall-through by default (opposite of C). Each case is implicitly broken. If you genuinely want fall-through, use the `fallthrough` keyword.
* A `switch` with no expression switches on `true` — useful for cleaner if-else chains:

```go
switch {
case x < 0:
    return "negative"
case x == 0:
    return "zero"
case x < 10:
    return "small"
default:
    return "large"
}
```

* The type switch (`switch v := x.(type)`) we saw above.

### `defer`

`defer` schedules a function call to run when the surrounding function returns, regardless of whether the return is normal or via panic:

```go
func readFile(name string) ([]byte, error) {
    f, err := os.Open(name)
    if err != nil {
        return nil, err
    }
    defer f.Close()    // runs when readFile returns

    return io.ReadAll(f)
}
```

Deferred calls run in LIFO order — last deferred runs first. Defer evaluates its arguments at the time of the `defer` statement, but the call happens at function return. Useful for:

* Closing files, connections, response bodies.
* Releasing mutexes (more in Lesson 2).
* Recovering from panics.

The `defer` pattern eliminates an entire class of "forgot to close the file" bugs. Use it heavily.

A subtle point: if you defer a method call on a struct, the receiver is captured at defer time. If the struct's value changes between `defer` and return, that change is reflected (because the receiver pointer points at the same struct).

## 1.9 Pointers (Briefly)

Go has pointers. They're simpler than C's:

* You can take the address of a variable: `p := &x`.
* You can dereference a pointer: `*p`.
* You cannot do pointer arithmetic.
* You cannot construct a pointer from an integer.
* The garbage collector follows pointers, so memory pointed to is kept alive.

```go
func modify(p *int) {
    *p = 42
}

func main() {
    x := 10
    modify(&x)
    fmt.Println(x)   // 42
}
```

When to use pointers:

* For function parameters when you want the function to mutate the caller's value.
* For struct receivers (as covered in 1.5).
* When passing big structs that you don't want copied.
* For optional values: a `*Foo` can be nil, signalling "no value."

When *not* to use pointers:

* Just because you can. Go is not C; you don't need pointers for ordinary variable manipulation.
* For slices, maps, channels, or interfaces — they're already reference-like internally.
* For premature optimisation — copying a small struct is fine.

### `new` and `&T{}`

Two ways to make a pointer to a new value:

```go
p1 := new(Order)         // pointer to zero-valued Order
p2 := &Order{ID: 1}      // pointer to Order with ID set
```

`&Order{...}` is much more common because it lets you set fields. `new()` is rare except in generic code.

(In Go 1.26, `new()` was extended to accept an initial value: `new(Order{ID: 1})`. Equivalent to `&Order{ID: 1}`. Mostly useful in generic functions.)

## 1.10 A Worked Example: An Order Book Skeleton

Let's tie 1.1 through 1.9 together. We'll build a small skeleton of an order book — not concurrent yet, just types and methods.

```go
package main

import (
    "errors"
    "fmt"
)

// Side represents the side of an order.
type Side int

const (
    Buy Side = iota
    Sell
)

func (s Side) String() string {
    switch s {
    case Buy:
        return "BUY"
    case Sell:
        return "SELL"
    default:
        return "UNKNOWN"
    }
}

// Order represents a limit order.
type Order struct {
    ID       int64
    Side     Side
    Price    int64    // in ticks (e.g., 1 tick = 1 cent)
    Quantity int64
}

// Notional returns the order's notional value in ticks.
func (o *Order) Notional() int64 {
    return o.Price * o.Quantity
}

// Book is a simple in-memory book of open orders.
type Book struct {
    orders map[int64]*Order
}

func NewBook() *Book {
    return &Book{
        orders: make(map[int64]*Order),
    }
}

// ErrOrderExists is returned when adding an order whose ID is already taken.
var ErrOrderExists = errors.New("order already exists")

// ErrOrderNotFound is returned when removing or looking up a missing order.
var ErrOrderNotFound = errors.New("order not found")

// Add inserts a new order. Returns ErrOrderExists if the ID is taken.
func (b *Book) Add(o *Order) error {
    if _, exists := b.orders[o.ID]; exists {
        return fmt.Errorf("add order %d: %w", o.ID, ErrOrderExists)
    }
    b.orders[o.ID] = o
    return nil
}

// Remove deletes an order by ID. Returns ErrOrderNotFound if missing.
func (b *Book) Remove(id int64) error {
    if _, exists := b.orders[id]; !exists {
        return fmt.Errorf("remove order %d: %w", id, ErrOrderNotFound)
    }
    delete(b.orders, id)
    return nil
}

// Get fetches an order by ID. Returns ErrOrderNotFound if missing.
func (b *Book) Get(id int64) (*Order, error) {
    o, exists := b.orders[id]
    if !exists {
        return nil, fmt.Errorf("get order %d: %w", id, ErrOrderNotFound)
    }
    return o, nil
}

// Size returns the number of open orders.
func (b *Book) Size() int {
    return len(b.orders)
}

func main() {
    book := NewBook()

    err := book.Add(&Order{ID: 1, Side: Buy, Price: 10000, Quantity: 5})
    if err != nil {
        fmt.Println(err)
        return
    }

    err = book.Add(&Order{ID: 2, Side: Sell, Price: 10100, Quantity: 3})
    if err != nil {
        fmt.Println(err)
        return
    }

    fmt.Printf("book has %d orders\n", book.Size())

    o, err := book.Get(1)
    if err != nil {
        fmt.Println(err)
        return
    }
    fmt.Printf("order 1: %s %d @ %d, notional %d\n", o.Side, o.Quantity, o.Price, o.Notional())

    // Try to add a duplicate.
    err = book.Add(&Order{ID: 1, Side: Buy, Price: 10000, Quantity: 5})
    if errors.Is(err, ErrOrderExists) {
        fmt.Println("can't add duplicate, as expected")
    }
}
```

What's at play:

* **Types as the spine of the API.** `Side`, `Order`, `Book` are the nouns. Methods on each describe the verbs.
* **Constructor function `NewBook()`.** Go has no constructors. Convention: `NewT()` for a function returning a `*T`. It returns a properly initialised Book — note we have to `make(map[...])` because the zero value of a map is nil and can't be written to.
* **Pointer receivers throughout.** `*Book` for methods; storing `*Order` in the map. Letting callers mutate orders through these pointers.
* **Sentinel errors plus wrapping.** `ErrOrderExists` is a package-level value. We wrap it in callsite-specific context with `fmt.Errorf("%w", ...)`. Callers can still test with `errors.Is(err, ErrOrderExists)`.
* **`String()` method on `Side`.** Satisfies `fmt.Stringer`, so `Printf("%s", o.Side)` works without us doing anything else.

Twenty lines of language gives us a usable mini-API. This is what idiomatic Go looks like at its core.

## 1.11 Style and Tooling You Adopt Day One

Three commands to internalise on day one. You should run them on every Go file you write.

* `go fmt ./...` — formats your code. There is one canonical style; `go fmt` enforces it. There are no debates. There is no `.gofmt.cfg`. Indentation is tabs (yes, tabs); spaces around operators are auto-managed; struct fields align in literals. Just run it. Most editors do this on save automatically.
* `go vet ./...` — runs static analysis. Catches things like format-string mismatches, suspicious assignments, struct tag typos. Run it before commit; CI should run it.
* `go test ./...` — runs all tests. We'll cover testing in Lesson 3.

There's also `golangci-lint` (a third-party umbrella linter) which runs dozens of additional checks. Most production teams use it.

Naming conventions:

* **CamelCase**, not snake_case. `MyFunction`, not `my_function`.
* **Capitalised = exported.** `Print` is exported; `print` is package-private. Same for fields, types, constants, methods.
* **Short names in small scopes.** `for i := range items` not `for index := range items`. Single-letter receivers (`b *Book`, `o *Order`).
* **Acronyms are CAPITAL.** `URL`, `ID`, `HTTP`. So `userID` or `UserID`, not `userId`.
* **Package names are lowercase, single-word.** `net`, `http`, `json`. No camel case.
* **Don't stutter.** `http.HTTPClient` is bad; the `http.` prefix already says it; just `http.Client`.

## 1.12 Summary: The Rules

1. **Static typing, no implicit conversions.** Be explicit about types.
2. **Every type has a zero value.** Variables are usable immediately after declaration.
3. **Multiple return values, especially `(T, error)`.** Almost every function that can fail returns an error.
4. **`if err != nil` is the rhythm.** No exceptions. Errors are values.
5. **Wrap errors with `fmt.Errorf("...: %w", err)`. Inspect with `errors.Is` / `errors.As`.**
6. **Structs are records. Methods are functions with a receiver.**
7. **Use pointer receivers consistently.** Pick one style per type.
8. **No classes. No inheritance. Use composition (embedding) or interfaces.**
9. **Interfaces are implicit and small.** A type satisfies an interface by having the right methods.
10. **`defer` for cleanup.** Files, locks, connections — close on the next line after opening.
11. **`go fmt`, `go vet`, `go test` are mandatory daily tools.**
12. **Capitalised names are exported. Lowercase are private to the package.**

# Lesson 2: Goroutines, Channels, and the Concurrency Model

## 2.1 Why This Lesson Exists

Most programming languages bolt concurrency onto a sequential core. Java adds threads and locks. Python adds threads, then asyncio when threads turned out to be problematic for I/O-bound work. JavaScript starts single-threaded and bolts on async/await. C# adds tasks and async/await. Each of these languages now has multiple ways to express concurrency, and the idiomatic choice depends on the workload.

Go's bet is different: **make concurrency a first-class part of the language and standardise on one mechanism**. That mechanism is goroutines and channels, based on Tony Hoare's 1978 paper *Communicating Sequential Processes* (CSP). The pitch: lightweight processes that communicate by passing messages, with the runtime doing all the heavy lifting. The slogan that sums it up:

> **Don't communicate by sharing memory; share memory by communicating.**

The traditional model (Java, C++, Python with threads): threads share memory, you protect shared memory with locks, you debug deadlocks and race conditions for years.

The CSP model (Go, Erlang, occam): processes don't share memory; they pass values to each other through channels. You compose programs out of producers and consumers connected by channels. Race conditions disappear because there's no shared state to race over.

In practice Go is pragmatic — it does have shared memory and mutexes when you need them — but the *first* tool you reach for is a channel. This is the most distinctive thing about Go, and it's why companies pick Go for backend services that need lots of concurrent I/O.

This lesson covers:

* **Goroutines** — what they are, how cheap they are, when to use them.
* **Channels** — the basic operations, blocking, buffered vs unbuffered, closing.
* **`select`** — choose between multiple channel operations.
* **The standard concurrency patterns** — fan-out / fan-in, worker pools, pipelines, timeouts.
* **The synchronisation primitives** — `sync.Mutex`, `sync.WaitGroup`, `sync.Once`, atomic.
* **The data race rules** — what's legal, what's not, the race detector.
* **`context.Context`** — Go's standard mechanism for cancellation and deadlines.

This is the densest lesson in the course. Take it slowly. Run every example. The drills will make you build a small but real concurrent program; you can't bluff this material.

## 2.2 Goroutines: Concurrency Made Cheap

A goroutine is a function running concurrently with other goroutines, in the same address space. You start one with the `go` keyword:

```go
package main

import (
    "fmt"
    "time"
)

func say(s string) {
    for i := 0; i < 3; i++ {
        fmt.Println(s)
        time.Sleep(100 * time.Millisecond)
    }
}

func main() {
    go say("world")    // goroutine
    say("hello")       // running in main goroutine
}
```

Output is interleaved:

```
hello
world
hello
world
hello
world
```

(The exact interleaving may vary.)

`go say("world")` starts a new goroutine running `say("world")`. The main function continues immediately to call `say("hello")`. The two run concurrently. When `main` returns, the program exits — *even if other goroutines are still running*. This is a frequent gotcha; we'll cover synchronisation in a moment.

### Goroutines are not threads

A goroutine is *not* an OS thread. The Go runtime multiplexes potentially millions of goroutines onto a small number of OS threads (typically one per CPU core). Goroutines start with about 2 KB of stack (vs ~8 MB for an OS thread); they can grow as needed. Switching between goroutines is faster than switching threads — no kernel involvement, just a runtime-level context save.

Concretely:

* You can have **millions** of goroutines in a single Go program. A typical web server handling 100,000 concurrent connections uses 100,000 goroutines, one per connection. This is normal.
* Starting a goroutine costs ~1 microsecond and ~2 KB of memory. Cheap enough to use freely.
* Goroutines are scheduled cooperatively by the Go runtime. The scheduler preempts long-running goroutines (since Go 1.14) and yields at function calls and channel operations.

This is fundamentally different from Java's threads. In Java, threads are a precious resource — you create a thread pool of size N (where N ≈ number of cores) and dispatch work to it. In Go, you spawn a goroutine for every unit of concurrent work without thinking about pools. The runtime manages the actual thread count.

### What goroutines are good for

* **I/O-bound work.** A web server with 10,000 concurrent connections uses 10,000 goroutines. While each is waiting for I/O, the scheduler runs others. Throughput scales with concurrency.
* **Pipelines and parallel data processing.** Producer goroutines feed consumer goroutines via channels.
* **Asynchronous tasks.** Send a request and don't wait for the response.

What they're *not* particularly good for:

* **CPU-bound work that doesn't parallelise.** A goroutine doesn't speed up sequential computation.
* **Saturating CPU cores with parallel computation.** Goroutines work for this, but the gains are bounded by the number of cores. If you have 8 cores, 1000 CPU-bound goroutines won't be faster than 8.

### Synchronisation: the simplest case

The "main exits before goroutines finish" problem above is solved with `sync.WaitGroup` (or with channels — coming next):

```go
import (
    "fmt"
    "sync"
)

func main() {
    var wg sync.WaitGroup

    wg.Add(3)
    for i := 1; i <= 3; i++ {
        i := i   // important: capture the loop variable
        go func() {
            defer wg.Done()
            fmt.Println("worker", i)
        }()
    }
    wg.Wait()
}
```

`WaitGroup`:

* `Add(n)` increments a counter.
* `Done()` decrements it (often via `defer`).
* `Wait()` blocks until the counter reaches zero.

Note the `i := i` trick. In Go 1.22+, the loop variable is per-iteration by default, so this is no longer necessary — but you'll still see it in older code, and you should know what it's doing. (Without `i := i`, all three goroutines see the same shared `i`, which by the time they run might be 4. Classic concurrency bug.)

## 2.3 Channels: The Basic Operations

A channel is a typed conduit. You send values into it from one goroutine; you receive them from another. The channel synchronises the goroutines.

```go
ch := make(chan int)        // unbuffered channel of int
ch <- 42                    // send (blocks until received)
v := <-ch                   // receive (blocks until sent)
v, ok := <-ch               // receive; ok=false if channel closed
close(ch)                   // close the channel
```

The key insight: an unbuffered channel forces send and receive to *meet*. A goroutine that sends blocks until another goroutine receives. A receiver blocks until a sender sends. This rendezvous is the synchronisation.

A complete example:

```go
package main

import "fmt"

func produce(ch chan<- int) {       // chan<- means send-only
    for i := 0; i < 5; i++ {
        ch <- i
    }
    close(ch)
}

func main() {
    ch := make(chan int)
    go produce(ch)

    for v := range ch {              // receive until channel closes
        fmt.Println(v)
    }
}
```

Output: `0 1 2 3 4`.

Walk through what happens:

1. Main creates an unbuffered channel `ch` and spawns a goroutine running `produce`.
2. `produce` tries `ch <- 0`. Blocks. (No receiver yet.)
3. Main starts the `for v := range ch` loop, which receives. Receives 0.
4. `produce` unblocks, loops, tries `ch <- 1`. Blocks.
5. Main's loop receives 1.
6. ... and so on through 4.
7. `produce` calls `close(ch)`.
8. Main's `range` sees the channel close and exits the loop.
9. Main's function ends; program exits.

The two goroutines are synchronised by the channel. The producer is never more than one send ahead of the consumer.

### Send-only and receive-only types

`chan int` is bidirectional. `chan<- int` is send-only; `<-chan int` is receive-only. You can convert bidirectional to either, but not the other way. This is used in function signatures to document direction:

```go
// Producer takes a channel it can only send to.
func produce(ch chan<- int) { ... }

// Consumer takes a channel it can only receive from.
func consume(ch <-chan int) { ... }
```

### Buffered channels

`make(chan int, 5)` creates a buffered channel with capacity 5. Sends don't block until the buffer is full; receives don't block until the buffer is empty.

```go
ch := make(chan int, 3)
ch <- 1   // doesn't block (buffer: [1])
ch <- 2   // doesn't block (buffer: [1, 2])
ch <- 3   // doesn't block (buffer: [1, 2, 3])
ch <- 4   // BLOCKS — buffer full
```

Buffered channels are useful as queues — but use them sparingly. The default should be unbuffered. Buffered channels often hide bugs by absorbing brief mismatches between producer and consumer rates; the bug surfaces only when the rates diverge for longer. Unbuffered channels make the synchronisation explicit.

When a buffered channel is *appropriate*:

* You know the exact number of items (e.g., spawning N workers, channel size N for results).
* The producer and consumer have known different rates and you want to smooth out bursts.
* Performance: the channel is a hot path and you've measured that buffering helps.

### Closing channels

`close(ch)` signals "no more values will be sent." A few rules:

* Only the sender should close. Closing twice or sending to a closed channel panics.
* Receivers can still read remaining buffered values after close.
* `v, ok := <-ch` returns `ok = false` and `v = zero value` when the channel is closed and drained.
* `for v := range ch` exits when the channel closes.

The convention: only the sender closes, and it closes when it has no more to send. With multiple senders, this gets tricky — coordinate with another channel or a `sync.Once`.

### Channel vs `nil`

A `nil` channel blocks forever — sends and receives never complete. This is occasionally useful (in `select`, to disable an arm), and otherwise a bug.

```go
var ch chan int
ch <- 1   // blocks forever
<-ch      // blocks forever
```

## 2.4 The `select` Statement

`select` lets a goroutine wait on multiple channel operations simultaneously, picking whichever is ready first. It's like `switch`, but for channels.

```go
select {
case msg := <-ch1:
    fmt.Println("from ch1:", msg)
case msg := <-ch2:
    fmt.Println("from ch2:", msg)
case ch3 <- 42:
    fmt.Println("sent to ch3")
case <-time.After(1 * time.Second):
    fmt.Println("timed out")
}
```

Each `case` is a channel operation (send or receive). `select` blocks until at least one case is ready, then executes that case. If multiple cases are ready, it picks one *at random* (this matters!). If no cases are ready, it blocks until one becomes ready — unless there's a `default` case, in which case `default` runs immediately.

```go
// Non-blocking receive.
select {
case msg := <-ch:
    fmt.Println(msg)
default:
    fmt.Println("nothing ready")
}
```

`select` is the most powerful concurrency primitive in Go. It enables:

* **Timeouts.** `case <-time.After(d):` fires after duration `d`.
* **Cancellation.** `case <-ctx.Done():` (we'll see context shortly).
* **Multi-source consumption.** Read from any of several producers.
* **Non-blocking sends/receives.** With `default:` you don't block.

### A timeout pattern

```go
func fetchWithTimeout(url string) (string, error) {
    resultCh := make(chan string, 1)
    errCh := make(chan error, 1)

    go func() {
        result, err := slowFetch(url)
        if err != nil {
            errCh <- err
            return
        }
        resultCh <- result
    }()

    select {
    case r := <-resultCh:
        return r, nil
    case err := <-errCh:
        return "", err
    case <-time.After(5 * time.Second):
        return "", errors.New("timeout")
    }
}
```

Note the buffered channels (capacity 1). If the goroutine completes after the timeout fires, it would otherwise block forever trying to send into an unbuffered channel that nobody's reading. The buffer-of-1 lets the goroutine complete its send and exit cleanly. This is a common pattern.

## 2.5 The Standard Concurrency Patterns

Five patterns cover 90% of concurrent Go you'll write. Memorise them.

### Pattern 1: Fan-out

One source, many workers, all reading from the same channel:

```go
func main() {
    work := make(chan int, 100)

    // Start 10 workers.
    var wg sync.WaitGroup
    for w := 0; w < 10; w++ {
        wg.Add(1)
        go func(id int) {
            defer wg.Done()
            for job := range work {
                fmt.Printf("worker %d processing %d\n", id, job)
                time.Sleep(100 * time.Millisecond)
            }
        }(w)
    }

    // Send work.
    for i := 0; i < 50; i++ {
        work <- i
    }
    close(work)

    wg.Wait()
}
```

Each worker reads from `work` until it closes. The channel itself does the load-balancing — whichever worker is free gets the next job. The closure of the channel signals "no more work," and once all workers drain it, they exit and `wg.Wait()` unblocks.

This is *the* canonical worker pool in Go. Replace the printing with real work, adjust the worker count, you have a production-grade concurrent task processor in 20 lines.

### Pattern 2: Fan-in

Many sources, one consumer, all sending to the same channel:

```go
func merge(chans ...<-chan int) <-chan int {
    out := make(chan int)
    var wg sync.WaitGroup

    for _, ch := range chans {
        wg.Add(1)
        go func(c <-chan int) {
            defer wg.Done()
            for v := range c {
                out <- v
            }
        }(ch)
    }

    go func() {
        wg.Wait()
        close(out)
    }()

    return out
}
```

Each input channel has a goroutine forwarding its values to `out`. A separate goroutine waits for all forwarders to finish, then closes `out`. Callers can `range` over the merged channel and see values from any source.

### Pattern 3: Pipeline

Multiple stages, each its own goroutine, connected by channels:

```go
func generate(nums ...int) <-chan int {
    out := make(chan int)
    go func() {
        defer close(out)
        for _, n := range nums {
            out <- n
        }
    }()
    return out
}

func square(in <-chan int) <-chan int {
    out := make(chan int)
    go func() {
        defer close(out)
        for n := range in {
            out <- n * n
        }
    }()
    return out
}

func main() {
    nums := generate(2, 3, 4, 5)
    squared := square(nums)
    for s := range squared {
        fmt.Println(s)   // 4, 9, 16, 25
    }
}
```

Each stage:

* Has one input channel and one output channel.
* Reads until input closes.
* Closes output (with `defer close(out)`) when done.

You can compose pipelines arbitrarily. Each stage runs in its own goroutine, so the work is concurrent.

### Pattern 4: Cancellation via `done` channel

Sometimes you want to tell goroutines to stop early. The classic pattern:

```go
func worker(done <-chan struct{}, work <-chan int) {
    for {
        select {
        case <-done:
            return
        case job, ok := <-work:
            if !ok {
                return
            }
            // process job
            _ = job
        }
    }
}

func main() {
    done := make(chan struct{})
    work := make(chan int)

    go worker(done, work)

    // ... eventually:
    close(done)   // signals all listeners
}
```

`chan struct{}` is the idiomatic "signal" channel — `struct{}` is zero bytes, so the channel carries no data, just the closure event. Closing it is a one-shot broadcast: every goroutine selecting on `<-done` sees the closed channel and proceeds.

In modern Go (post-1.7), this pattern is largely subsumed by `context.Context`. We'll cover that in 2.9.

### Pattern 5: Bounded concurrency with a semaphore

Sometimes you want N goroutines max, but not exactly N. A buffered channel makes a perfect counting semaphore:

```go
sem := make(chan struct{}, 10)   // max 10 concurrent

for _, item := range items {
    item := item
    sem <- struct{}{}            // acquire (blocks if 10 in flight)
    go func() {
        defer func() { <-sem }() // release
        process(item)
    }()
}
```

Or use the `golang.org/x/sync/semaphore` package, which adds support for weighted acquisition.

## 2.6 Sync Primitives: When You Don't Want a Channel

Channels are great. They're not always the right tool. Sometimes you have shared state that multiple goroutines read and write, and a channel-based design is clumsier than a lock.

The `sync` package has the primitives.

### `sync.Mutex`

The classic mutual exclusion lock:

```go
type Counter struct {
    mu    sync.Mutex
    count int
}

func (c *Counter) Increment() {
    c.mu.Lock()
    defer c.mu.Unlock()
    c.count++
}

func (c *Counter) Value() int {
    c.mu.Lock()
    defer c.mu.Unlock()
    return c.count
}
```

Standard practice:

* Embed the mutex in the struct it protects.
* Use `defer c.mu.Unlock()` immediately after `Lock`. Guarantees unlock even on panic.
* Don't copy a struct that contains a mutex — `go vet` will catch this.

Note: Go's mutex is not reentrant. If a goroutine that already holds the mutex tries to lock it again, it deadlocks. Don't do that.

### `sync.RWMutex`

Read-write mutex: many readers OR one writer.

```go
type Cache struct {
    mu    sync.RWMutex
    items map[string]string
}

func (c *Cache) Get(key string) (string, bool) {
    c.mu.RLock()
    defer c.mu.RUnlock()
    v, ok := c.items[key]
    return v, ok
}

func (c *Cache) Set(key, value string) {
    c.mu.Lock()
    defer c.mu.Unlock()
    c.items[key] = value
}
```

Reach for `RWMutex` when reads heavily outnumber writes. For mostly-balanced workloads, plain `Mutex` is often faster (lower overhead per operation).

### `sync.WaitGroup`

Already covered. Tracks "are all my goroutines done yet?"

* `Add(delta)` adds to the counter.
* `Done()` decrements (equivalent to `Add(-1)`).
* `Wait()` blocks until counter is 0.

Always call `Add` *before* spawning the goroutine. Calling it inside the goroutine is racy.

### `sync.Once`

Run a function exactly once, even if called from multiple goroutines:

```go
var (
    once   sync.Once
    config *Config
)

func GetConfig() *Config {
    once.Do(func() {
        config = loadConfig()
    })
    return config
}
```

`once.Do(f)` runs `f` exactly once across all goroutines. Subsequent calls return immediately. Useful for lazy initialisation.

### `sync.Map`

A concurrent map optimised for specific access patterns (lots of reads, infrequent writes; or keys written once and read many times). Don't use it as a general-purpose concurrent map; use a regular `map[K]V` plus a `sync.RWMutex` instead. The cases where `sync.Map` is faster are narrow.

### Atomic operations

`sync/atomic` provides atomic operations on integers and pointers. Faster than mutexes for simple counters:

```go
import "sync/atomic"

var counter atomic.Int64

func increment() {
    counter.Add(1)
}

func value() int64 {
    return counter.Load()
}
```

The `atomic.Int64`, `atomic.Uint64`, `atomic.Pointer[T]`, etc., types (added in Go 1.19) are the modern idiom. For a simple counter accessed from many goroutines, atomic is faster than a mutex by 5-10x.

### When to use what

A useful decision tree:

* **Coordinating goroutine completion?** `sync.WaitGroup`.
* **One-time initialisation?** `sync.Once`.
* **Simple counter or flag?** `atomic`.
* **Shared state with reads and writes?** `sync.Mutex` or `sync.RWMutex`.
* **Producer-consumer or pipeline?** `chan`.
* **Multiple goroutines waiting for one event?** `chan` with `close()` as broadcast.
* **Cancellation, timeouts, deadlines?** `context.Context`.

The textbook Go advice — "use channels by default" — is mostly right but oversimplified. For shared state in a single struct, a mutex is often clearer than threading the state through channels. The Go authors themselves use mutexes throughout the standard library. Idiomatic Go is: channels for *communication*, mutexes for *state*.

## 2.7 The Data Race Problem

A **data race** is when two goroutines access the same memory, at least one of them is writing, and there's no synchronisation between them. Data races cause undefined behaviour. Your program might work today, fail tomorrow, produce wrong answers, or corrupt memory.

```go
// CLASSIC RACE — DO NOT DO THIS
var counter int

func increment() {
    counter++   // read counter, add 1, write counter — three operations
}

func main() {
    for i := 0; i < 1000; i++ {
        go increment()
    }
    time.Sleep(time.Second)
    fmt.Println(counter)   // expected 1000; actually some smaller number
}
```

`counter++` is *not atomic*. It's:

1. Read `counter` from memory.
2. Add 1.
3. Write `counter` back.

If two goroutines do this simultaneously, both can read the old value, both compute old+1, both write old+1 — net effect is one increment, not two. Updates are lost.

Worse: in a less benign case, the compiler and CPU are free to reorder memory operations across goroutines. You can observe weirdness like "x was set to 1, but another goroutine sees x as 0 and y as some new value" — order violations that look impossible.

Fixes:

```go
// 1. Mutex
var mu sync.Mutex
mu.Lock()
counter++
mu.Unlock()

// 2. Atomic
atomic.AddInt64(&counter, 1)

// 3. Channel: send increment requests to a single goroutine
incCh <- 1
```

### The race detector

Go ships with a built-in race detector. Run your tests or program with `-race`:

```sh
go test -race ./...
go run -race main.go
```

It instruments memory accesses and reports races at runtime when they occur. It's not free (programs run 2-10x slower under -race) but it's invaluable for finding real bugs. Run your tests under -race in CI.

The race detector is the gold standard. If `-race` says your code has a race, your code has a race. If `-race` doesn't flag a race, your code might still have one — but only if you've not exercised the racy path.

### The Go memory model

Go has a [formal memory model](https://go.dev/ref/mem) describing exactly when one goroutine's writes are visible to another. The summary:

* Writes by goroutine A are visible to goroutine B only if A and B synchronise via channels, mutexes, atomics, `WaitGroup`, etc.
* Without synchronisation, no ordering is guaranteed.
* Reading and writing values larger than a machine word is not atomic — even pointer assignment can be torn on some architectures.

You don't need to memorise the memory model in detail. The practical rules:

1. **Don't read or write shared memory without synchronisation.** Use mutexes, channels, or atomics.
2. **Use the race detector.** It'll catch most bugs.
3. **Don't rely on apparent atomicity of simple operations.** `x = y` for an `int` may or may not be atomic depending on the architecture. Use `atomic` if you need it.

## 2.8 Common Antipatterns

A tour of the things that go wrong.

### Forgetting to wait for goroutines

```go
func main() {
    go doWork()
    // main returns; doWork might be cut off mid-execution
}
```

Always synchronise: `WaitGroup`, channel, or `context`.

### Goroutine leaks

```go
func leak() {
    ch := make(chan int)
    go func() {
        v := <-ch    // blocks forever; channel never sent to
        fmt.Println(v)
    }()
    // goroutine is now stuck forever
}
```

Goroutines that block on a channel that will never receive a value leak. They consume a couple of KB each, plus whatever they reference. Long-running services with leaks can grow to gigabytes.

Detection: pprof goroutine profiles will show you stuck goroutines. Periodically check.

Prevention: ensure every goroutine has a way to exit. Use `context.Context` for cancellation. Always close channels when you're done sending.

### Incorrect closure of channel with multiple senders

```go
// WRONG: only one sender can close
for i := 0; i < 10; i++ {
    go func() {
        ch <- compute()
        close(ch)   // PANIC — second close
    }()
}
```

When you have multiple senders, the consumer should close — but the consumer often doesn't know when to. Patterns:

* Use `WaitGroup`. When all senders are done, *one* goroutine (often a coordinator) closes the channel.
* Use the `chan struct{}` done pattern; senders never close, they exit when `done` is signalled.

### Passing the loop variable into a goroutine (pre-Go 1.22)

```go
// PRE-1.22 BUG
for i := 0; i < 5; i++ {
    go func() {
        fmt.Println(i)
    }()
}
```

Before Go 1.22, all goroutines share `i`. They all print 5 (or some other near-final value). The fix was `i := i` to capture by copy. **Go 1.22 and later** make the loop variable per-iteration by default, so this works — but you'll see the `i := i` pattern in older code.

### Using `time.Tick` with no goroutine that exits

```go
ch := time.Tick(time.Second)   // returns a channel that ticks forever
// ... if the surrounding goroutine exits without reading from ch,
// the underlying timer leaks.
```

`time.Tick` is intended for "tick forever in main." For bounded use, use `time.NewTicker` and `defer ticker.Stop()`.

### Mutex copied by value

```go
type Counter struct {
    mu sync.Mutex
    n  int
}

func passByValue(c Counter) { ... }   // BUG: copies the mutex
```

Copying a mutex is a bug — the copy is a separate mutex, no longer protecting what you think it protects. Always pass `*Counter`, never `Counter`. `go vet` catches this.

## 2.9 Context: Cancellation, Deadlines, Values

`context.Context` is Go's standard mechanism for *cancellation*, *deadlines*, and *request-scoped values*. Every long-running function in modern Go takes a `context.Context` as its first argument.

```go
import "context"

func doWork(ctx context.Context) error {
    for {
        select {
        case <-ctx.Done():
            return ctx.Err()
        default:
            // do a unit of work
        }
    }
}
```

`ctx.Done()` returns a channel that's closed when the context is cancelled or expires. `ctx.Err()` tells you why (`context.Canceled` or `context.DeadlineExceeded`).

### Creating contexts

```go
// Top-level: a context that never cancels.
ctx := context.Background()

// Add a timeout: cancels after 5 seconds.
ctx, cancel := context.WithTimeout(ctx, 5*time.Second)
defer cancel()   // always defer cancel

// Add a deadline.
ctx, cancel = context.WithDeadline(ctx, time.Now().Add(5*time.Second))
defer cancel()

// Make it manually cancellable.
ctx, cancel = context.WithCancel(ctx)
defer cancel()
// ... cancel() can be called from elsewhere to abort.
```

Always `defer cancel()`. Even if the context expires naturally, calling cancel releases resources.

### Context propagation

The standard convention: `Context` is the *first* argument of any function that does I/O, calls into other services, or might run for a long time. Pass it through your call stack:

```go
func handleRequest(ctx context.Context, req *Request) error {
    user, err := loadUser(ctx, req.UserID)
    if err != nil {
        return err
    }
    return processOrder(ctx, user, req)
}

func loadUser(ctx context.Context, id int64) (*User, error) {
    return db.QueryUser(ctx, id)
}
```

When the top-level `ctx` is cancelled, every downstream operation that's checking `ctx.Done()` will exit. This is how you cancel a whole tree of operations at once — say, when an HTTP client disconnects.

`net/http`, `database/sql`, `os/exec`, all the third-party RPC libraries — all take `context.Context` and respect cancellation.

### Context values

You can attach values to a context with `context.WithValue`. This is meant for *request-scoped* metadata — request IDs, user identity, tracing spans.

```go
ctx = context.WithValue(ctx, requestIDKey{}, "abc123")

// Later:
if id, ok := ctx.Value(requestIDKey{}).(string); ok {
    log.Println("request:", id)
}
```

Don't use this for ordinary parameters. The convention is "metadata that should travel with a request through many layers, that intermediate code shouldn't have to know about." Logging IDs, auth tokens, tracing — yes. Function arguments — no.

### When NOT to use context

* As a struct field: `Context` should be passed explicitly to functions, not stored in long-lived structs. (Some exceptions exist for stateful long-running things, but the default is to pass.)
* For optional parameters: that's not what it's for. Use plain function arguments.
* Without a deadline or cancellation source if you'll never cancel: just use `context.Background()` and document why.

## 2.10 A Worked Example: A Concurrent Web Crawler

Let's tie 2.1-2.9 together. We'll build a small concurrent crawler that fetches URLs and extracts links, with bounded concurrency, timeouts, and cancellation.

```go
package main

import (
    "context"
    "fmt"
    "io"
    "net/http"
    "regexp"
    "strings"
    "sync"
    "time"
)

type Crawler struct {
    client      *http.Client
    concurrency int
    visited     sync.Map   // url -> struct{}
}

func NewCrawler(concurrency int) *Crawler {
    return &Crawler{
        client:      &http.Client{Timeout: 5 * time.Second},
        concurrency: concurrency,
    }
}

// Result is one fetched page.
type Result struct {
    URL   string
    Links []string
    Err   error
}

// Crawl fetches the seed URL and follows links breadth-first, bounded by depth.
// Returns a channel of results; closes it when crawling is complete.
func (c *Crawler) Crawl(ctx context.Context, seed string, maxDepth int) <-chan Result {
    results := make(chan Result, c.concurrency)
    sem := make(chan struct{}, c.concurrency)

    var wg sync.WaitGroup

    var crawl func(url string, depth int)
    crawl = func(url string, depth int) {
        defer wg.Done()

        if depth > maxDepth {
            return
        }
        if _, seen := c.visited.LoadOrStore(url, struct{}{}); seen {
            return
        }

        // Acquire a slot.
        select {
        case sem <- struct{}{}:
        case <-ctx.Done():
            return
        }
        defer func() { <-sem }()

        links, err := c.fetch(ctx, url)
        select {
        case results <- Result{URL: url, Links: links, Err: err}:
        case <-ctx.Done():
            return
        }

        if err != nil {
            return
        }

        for _, link := range links {
            wg.Add(1)
            go crawl(link, depth+1)
        }
    }

    wg.Add(1)
    go crawl(seed, 0)

    // When all goroutines done, close results.
    go func() {
        wg.Wait()
        close(results)
    }()

    return results
}

var linkRe = regexp.MustCompile(`href="(http[^"]+)"`)

func (c *Crawler) fetch(ctx context.Context, url string) ([]string, error) {
    req, err := http.NewRequestWithContext(ctx, "GET", url, nil)
    if err != nil {
        return nil, err
    }
    resp, err := c.client.Do(req)
    if err != nil {
        return nil, err
    }
    defer resp.Body.Close()

    body, err := io.ReadAll(resp.Body)
    if err != nil {
        return nil, err
    }

    var links []string
    for _, m := range linkRe.FindAllStringSubmatch(string(body), -1) {
        link := strings.TrimSpace(m[1])
        if link != "" {
            links = append(links, link)
        }
    }
    return links, nil
}

func main() {
    ctx, cancel := context.WithTimeout(context.Background(), 30*time.Second)
    defer cancel()

    c := NewCrawler(10)
    results := c.Crawl(ctx, "https://example.com", 2)

    for r := range results {
        if r.Err != nil {
            fmt.Printf("ERR %s: %v\n", r.URL, r.Err)
            continue
        }
        fmt.Printf("OK  %s (%d links)\n", r.URL, len(r.Links))
    }
}
```

Read the patterns at play:

* **Bounded concurrency via a semaphore channel** (`sem`).
* **Visited-URL tracking via `sync.Map`** (a case where it's appropriate: many writes of unique keys, no key reuse).
* **`WaitGroup` for "all crawls done"**, with a goroutine to close the results channel afterward.
* **`context.Context` for timeout and cancellation.** If the top-level context expires (30 seconds), every goroutine's `select` on `ctx.Done()` fires, and they exit cleanly.
* **Streaming results via channel.** The caller reads results as they arrive, not all at once.

Real production crawlers add: politeness (`robots.txt`, rate limiting per domain), retry logic, persistent storage, structured link extraction (HTML parser, not regex). But the concurrency skeleton is what's shown here.

## 2.11 Summary: The Rules

1. **Goroutines are cheap.** Spawn freely.
2. **`go func()` doesn't synchronise.** Use `WaitGroup`, channel, or context to wait.
3. **Channels rendezvous unbuffered, queue when buffered.** Default to unbuffered.
4. **`for v := range ch` reads until close.**
5. **Close from the sender side; only one sender; close once.**
6. **`select` for multiple channel operations.** Random selection on ties.
7. **`time.After` and `ctx.Done()` integrate timeouts and cancellation into select.**
8. **The five canonical patterns: fan-out, fan-in, pipeline, cancellation, semaphore.**
9. **"Don't communicate by sharing memory; share memory by communicating."** But mutexes are fine for shared state.
10. **`sync.Mutex` is not reentrant.** Don't lock twice in the same goroutine.
11. **Always `defer cancel()` after `WithTimeout` / `WithCancel`.**
12. **Pass `context.Context` as the first argument to long-running and I/O functions.**
13. **Run tests with `-race`.** Catch data races early.
14. **Goroutines that block forever leak.** Audit cancellation paths.
15. **`go vet` catches mutex copies, format string mismatches, and other common bugs.** Run it.

Implement a function `Process` that takes:

## 3.2 Slices: The Most Misunderstood Type in Go

Every Go programmer's first month includes a moment when they assign a slice to another variable, modify one of them, and find that the other has changed too. They were not prepared for this. Let's prepare you.

### Arrays first

Go has *arrays* — fixed-size, value-typed sequences:

```go
var a [3]int        // [0, 0, 0]
b := [3]int{1, 2, 3}
c := [...]int{1, 2, 3, 4}   // size inferred: [4]int

// Arrays are values. Assignment copies.
d := b   // d is [3]int{1, 2, 3}
d[0] = 99
fmt.Println(b)   // [1 2 3] — unchanged
fmt.Println(d)   // [99 2 3]
```

Arrays are rare in idiomatic Go. The size is part of the type — `[3]int` and `[4]int` are different types — which makes them inflexible. You'll see arrays for fixed-size lookup tables, hash digests, and inside the implementations of slices.

### Slices are descriptors

A slice is a struct with three fields:

```go
type SliceHeader struct {
    Data uintptr   // pointer to the backing array
    Len  int       // number of elements visible
    Cap  int       // capacity of the backing array (from Data forward)
}
```

A slice doesn't own data; it points at an array. Multiple slices can share a backing array. This is the source of the aliasing trap.

Visually:

```
slice1 := []int{1, 2, 3, 4, 5}

  slice1: { Data: &arr[0], Len: 5, Cap: 5 }
            │
            ▼
  arr:   [1, 2, 3, 4, 5]
```

`make([]T, n)` creates a slice of length `n` (and cap `n`) backed by a fresh array.

`make([]T, n, c)` creates a slice of length `n` and capacity `c`.

```go
s := make([]int, 3, 10)   // len 3, cap 10
fmt.Println(len(s), cap(s))   // 3 10
```

### Slicing creates aliases

The `[i:j]` syntax produces a new slice viewing the same backing array:

```go
a := []int{0, 1, 2, 3, 4, 5}
b := a[1:4]   // [1, 2, 3]
b[0] = 99
fmt.Println(a)   // [0 99 2 3 4 5] — a was modified!
```

Both `a` and `b` point at the same backing array. Modifying `b[0]` modifies `a[1]`. This is *the* gotcha.

### Append: the deeper gotcha

`append` grows a slice. If capacity allows, it writes in place:

```go
s := make([]int, 0, 10)
s = append(s, 1, 2, 3)   // length 3, cap 10; no reallocation
```

If capacity isn't enough, append allocates a new bigger backing array and copies:

```go
s := []int{1, 2, 3}      // len 3, cap 3
s = append(s, 4)         // cap exceeded; new array allocated
```

The two behaviors look the same from the call site but have wildly different aliasing consequences:

```go
a := []int{1, 2, 3, 4, 5}
b := a[:3]            // [1, 2, 3]; shares backing array with a
b = append(b, 99)     // does this go into a's backing array?
fmt.Println(a)        // depends on whether append reused or reallocated!
```

`b` had `len 3, cap 5` (capacity from the start of the backing array, which is shared). `append` sees cap 5, so writes the 99 at index 3 of the *shared* array. Now `a` is `[1, 2, 3, 99, 5]`. You probably didn't want that.

If `b` had been a copy (`b := append([]int(nil), a[:3]...)`), or if `a`'s capacity had been exhausted, the append would have allocated a fresh array and `a` would be untouched.

The rule: **`append` may or may not reallocate. Don't rely on aliasing through `append`.** When in doubt, copy:

```go
b := make([]int, 3)
copy(b, a[:3])
// or:
b := append([]int(nil), a[:3]...)
// or (Go 1.21+):
b := slices.Clone(a[:3])
```

Then mutations to `b` and `b = append(b, x)` won't ever affect `a`.

### Common slice operations

```go
// Length
n := len(s)

// Append
s = append(s, x)
s = append(s, x, y, z)
s = append(s, otherSlice...)   // unpack

// Copy
n := copy(dst, src)   // copies min(len(dst), len(src))

// Remove element at index i (preserve order)
s = append(s[:i], s[i+1:]...)

// Remove element at index i (don't care about order; faster)
s[i] = s[len(s)-1]
s = s[:len(s)-1]

// Insert element at index i
s = append(s[:i], append([]T{value}, s[i:]...)...)
// Or, less elegant but more efficient:
s = append(s, *new(T))         // grow by one
copy(s[i+1:], s[i:])           // shift right
s[i] = value
```

The "remove preserving order" pattern is so common it's worth memorising.

### `slices` package (Go 1.21+)

Modern Go has a `slices` package with generic helpers:

```go
import "slices"

slices.Contains([]int{1, 2, 3}, 2)            // true
slices.Index([]int{1, 2, 3}, 2)               // 1
slices.Sort([]int{3, 1, 2})                   // sorts in-place
slices.SortFunc(orders, func(a, b Order) int { return cmp.Compare(a.Price, b.Price) })
slices.Reverse([]int{1, 2, 3})
slices.Clone(s)
slices.Equal(a, b)
slices.Insert(s, 2, 99)
slices.Delete(s, 1, 3)   // remove indices 1..3
```

These didn't exist for a long time, so older code reimplements them everywhere. New code should reach for `slices`.

### Performance note: pre-allocate when you know the size

```go
// Slow: many reallocations
var s []int
for i := 0; i < 1_000_000; i++ {
    s = append(s, i)
}

// Fast: pre-allocate
s := make([]int, 0, 1_000_000)
for i := 0; i < 1_000_000; i++ {
    s = append(s, i)
}
```

`append` with a too-small capacity allocates new backing arrays repeatedly. Each reallocation copies all existing elements. The growth factor is around 2x for small slices, 1.25x for large ones, so the total cost amortises to O(n) — but the constants matter, and for known-size workloads, pre-allocating wins.

## 3.3 Maps

A `map[K]V` is a hash table. Keys must be comparable (`==` works on them). Most basic types qualify; slices and maps don't. Structs do, if all their fields are comparable.

```go
m := make(map[string]int)
m["one"] = 1
m["two"] = 2

m2 := map[string]int{
    "three": 3,
    "four":  4,
}

// Read
v := m["one"]                     // 1
missing := m["missing"]           // 0 (zero value), no error
v, ok := m["maybe"]               // ok is false if absent

// Delete
delete(m, "one")

// Iterate (order is RANDOM)
for k, v := range m {
    fmt.Println(k, v)
}

// Length
n := len(m)
```

### The nil map gotcha

A `nil` map can be *read* from (returns zero value) but *can't be written to* — that panics:

```go
var m map[string]int
fmt.Println(m["key"])    // OK: 0
m["key"] = 1             // PANIC: assignment to entry in nil map
```

Always `make` your maps before writing. Or use a literal: `m := map[string]int{}`.

### Random iteration order

The `range` over a map gives keys in *unspecified* (and randomised) order. Two iterations of the same map can yield different orders. This is intentional — Go developers don't want you accidentally depending on insertion order.

If you need deterministic order (logging, output, hashing, deterministic tests), sort the keys:

```go
keys := make([]string, 0, len(m))
for k := range m {
    keys = append(keys, k)
}
sort.Strings(keys)
for _, k := range keys {
    fmt.Println(k, m[k])
}
```

In Go 1.23+, you can iterate via `maps.Keys`:

```go
import "slices"
import "maps"

keys := slices.Sorted(maps.Keys(m))
for _, k := range keys {
    fmt.Println(k, m[k])
}
```

### Concurrent maps

Built-in maps are *not* safe for concurrent use. If two goroutines write to the same map simultaneously, the runtime detects it and crashes the program with "fatal error: concurrent map writes." This is a deliberate fail-fast.

Solutions:

* Wrap with a `sync.RWMutex`.
* Use `sync.Map` if your access pattern matches what it's optimised for.
* Use a channel to serialise access via a single owner goroutine.

```go
type SafeMap struct {
    mu sync.RWMutex
    m  map[string]int
}

func (s *SafeMap) Get(k string) (int, bool) {
    s.mu.RLock()
    defer s.mu.RUnlock()
    v, ok := s.m[k]
    return v, ok
}

func (s *SafeMap) Set(k string, v int) {
    s.mu.Lock()
    defer s.mu.Unlock()
    s.m[k] = v
}
```

This is fine for the vast majority of cases.

## 3.4 Strings, Runes, Bytes

Go strings are **immutable byte sequences**. They're typically UTF-8 encoded but the language doesn't enforce this — a string is just bytes.

```go
s := "héllo"           // UTF-8 bytes: 68 c3 a9 6c 6c 6f (6 bytes)
fmt.Println(len(s))    // 6, not 5! len returns byte count
fmt.Println(s[0])      // 104 ('h' as a byte)
fmt.Println(s[1])      // 195 (first byte of é, NOT é itself)
```

Indexing a string gives a byte, not a "character." This trips up everyone the first time.

For character-level access, you have to decode UTF-8 to *runes* (Unicode code points):

```go
s := "héllo"
for i, r := range s {
    fmt.Printf("byte index %d: rune %c (%d)\n", i, r, r)
}
// byte index 0: rune h (104)
// byte index 1: rune é (233)
// byte index 3: rune l (108)
// byte index 4: rune l (108)
// byte index 5: rune o (111)
```

`for i, r := range s` decodes UTF-8 on the fly. `i` is the byte index of where each rune starts; `r` is the rune (`int32`).

To get a `[]rune` (slice of code points):

```go
runes := []rune("héllo")
fmt.Println(len(runes))   // 5
fmt.Println(string(runes[1]))   // "é"
```

`[]byte("...")` gets you the raw bytes. `string([]byte{...})` converts back. These conversions are *copies*, not aliases — strings are immutable, so it can't be otherwise.

### String building

Concatenating with `+` allocates a new string each time. For lots of concatenation, use `strings.Builder`:

```go
var b strings.Builder
for i := 0; i < 1000; i++ {
    b.WriteString("hello ")
}
result := b.String()
```

Or, when you have all parts in advance:

```go
parts := []string{"hello", "world", "!"}
result := strings.Join(parts, " ")
```

Or `fmt.Sprintf` for formatted output (slower but flexible).

### Useful packages

* `strings` — `Contains`, `HasPrefix`, `HasSuffix`, `Split`, `Join`, `ReplaceAll`, `ToLower`, `Trim`, `Builder`.
* `strconv` — number parsing/formatting: `Atoi`, `Itoa`, `ParseFloat`, `FormatInt`.
* `unicode` — character classification: `IsDigit`, `IsLetter`, `ToUpper`.
* `unicode/utf8` — UTF-8 encoding/decoding details.
* `regexp` — regular expressions, RE2 syntax (no backreferences; linear-time guarantees).

## 3.5 Packages and Modules

### Packages

A package is a directory of `.go` files that share a `package` declaration.

```
myproject/
├── go.mod
├── main.go              # package main
└── orders/
    ├── order.go         # package orders
    ├── book.go          # package orders
    └── book_test.go     # package orders
```

To use code from another package, import it:

```go
package main

import (
    "fmt"
    "myproject/orders"
)

func main() {
    b := orders.NewBook()
    fmt.Println(b.Size())
}
```

The capital-letter rule: only exported (capitalised) names from another package are visible.

Common conventions:

* One concept per package. `orders` for orders; `users` for users; not `model` or `util` (those are package smells).
* The package name is the last element of the import path (`net/http` → `http`), unless the file declares a different name (rare; mostly for collision avoidance).
* The `_test.go` suffix is special — those files are only compiled when running tests. They can be in the same package (white-box testing) or a `mypackage_test` package (black-box testing).

### Modules

A *module* is a collection of packages versioned together. It has a `go.mod` file at its root:

```
// go.mod
module github.com/yourname/myproject

go 1.26

require (
    github.com/some/dependency v1.2.3
)
```

To start a new module: `go mod init github.com/yourname/myproject`. The module path is typically the URL where it's hosted, but for private code it can be anything.

Common workflow commands:

```sh
go mod init github.com/yourname/myproject   # create go.mod
go get github.com/some/dependency           # add a dependency
go get -u                                   # update all dependencies
go mod tidy                                 # clean up: remove unused, add missing
go mod download                             # download all to local cache
go build ./...                              # build everything
go test ./...                               # test everything
```

The `./...` pattern means "this directory and all subdirectories recursively." Use it for project-wide commands.

### Versioning

Module versions follow semver: `v1.2.3`. The version `v0.x.x` indicates "unstable; breaking changes may happen." Modules at `v2` or higher have their major version in the import path: `github.com/foo/bar/v2`. This is unusual but explicit.

Dependencies are recorded in `go.mod` with their exact version. The actual content hashes are recorded in `go.sum` — commit both files. Go's module system verifies the hashes on download to catch tampering.

### Internal packages

A package under a directory called `internal/` is only importable by code at or below the directory's parent. So `myproject/internal/...` can only be imported by `myproject/...` — not by other projects. Use this to expose intentional public API only:

```
myproject/
├── go.mod
├── public/
│   └── api.go            # exported, others can import
└── internal/
    └── magic/
        └── helpers.go    # only myproject/* can import
```

A common pattern: put most of your code in `internal/`, expose a thin public API.

### Vendoring (rare)

`go mod vendor` copies all your dependencies into a `vendor/` directory inside your project. Builds use the vendored copies. Useful for reproducible builds without network access (CI environments, air-gapped). Increasingly rare with the module proxy and reliable hashing.

## 3.6 Testing

Go has built-in testing. No external framework needed.

### A test file

Save next to the code it tests, named `xxx_test.go`:

```go
// orders/book_test.go
package orders

import (
    "testing"
)

func TestNewBook_StartsEmpty(t *testing.T) {
    b := NewBook()
    if got := b.Size(); got != 0 {
        t.Errorf("Size() = %d, want 0", got)
    }
}

func TestBook_AddDuplicate(t *testing.T) {
    b := NewBook()
    o := &Order{ID: 1}

    if err := b.Add(o); err != nil {
        t.Fatalf("Add() unexpected error: %v", err)
    }

    err := b.Add(o)
    if err == nil {
        t.Fatal("expected error on duplicate Add, got nil")
    }
    if !errors.Is(err, ErrOrderExists) {
        t.Errorf("err = %v, want %v", err, ErrOrderExists)
    }
}
```

Run with `go test ./...` (or `go test ./orders/` for one package).

The test functions:

* Start with `Test`, take `*testing.T`.
* Use `t.Errorf(...)` to record a failure but continue.
* Use `t.Fatalf(...)` to fail and stop this test.
* Test name conventions: `TestThing_DoesSomething` is common.

### Table-driven tests

The idiomatic pattern for testing many cases:

```go
func TestNotional(t *testing.T) {
    cases := []struct {
        name     string
        order    Order
        expected int64
    }{
        {"zero quantity", Order{Price: 100, Quantity: 0}, 0},
        {"normal", Order{Price: 100, Quantity: 5}, 500},
        {"large", Order{Price: 1_000_000, Quantity: 1_000_000}, 1_000_000_000_000},
    }

    for _, c := range cases {
        t.Run(c.name, func(t *testing.T) {
            got := c.order.Notional()
            if got != c.expected {
                t.Errorf("Notional() = %d, want %d", got, c.expected)
            }
        })
    }
}
```

`t.Run` creates a subtest. Each row of the table becomes a test you can run individually with `go test -run TestNotional/normal`.

### Benchmarks

```go
func BenchmarkBookAdd(b *testing.B) {
    book := NewBook()
    b.ResetTimer()
    for i := 0; i < b.N; i++ {
        book.Add(&Order{ID: int64(i)})
    }
}
```

`b.N` is set by the testing framework to whatever number gets a stable measurement. Run with:

```sh
go test -bench=. -benchmem ./orders/
```

`-benchmem` reports allocations per iteration. Critical for performance work.

### Test coverage

```sh
go test -cover ./...
go test -coverprofile=cover.out ./...
go tool cover -html=cover.out
```

The HTML view highlights which lines were exercised. Useful for finding untested branches.

### Examples (executable docs)

```go
func ExampleNewBook() {
    b := NewBook()
    b.Add(&Order{ID: 1, Price: 100, Quantity: 5})
    fmt.Println(b.Size())
    // Output: 1
}
```

`Example` functions are in test files but appear in `go doc` output. The `// Output:` comment is verified during `go test` — if the function's stdout doesn't match, the test fails. They're documentation that can't go stale.

### Fuzzing

Go has built-in fuzz testing (since Go 1.18):

```go
func FuzzParseInt(f *testing.F) {
    f.Add("42")
    f.Add("-1")
    f.Add("0")
    f.Fuzz(func(t *testing.T, s string) {
        n, err := strconv.Atoi(s)
        if err == nil {
            // round trip should preserve
            if got := strconv.Itoa(n); got != strings.TrimLeft(s, "+0") {
                t.Errorf("round trip lost info: %q -> %d -> %q", s, n, got)
            }
        }
    })
}
```

Run with `go test -fuzz=FuzzParseInt`. The runtime feeds increasingly complex inputs; if it finds one that fails, it minimises and reports.

## 3.7 The Standard Library Tour

Go's standard library is huge and mostly excellent. Know what's there before you reach for a third-party crate.

### Core

* **`fmt`** — formatted I/O. `Print`, `Printf`, `Println`, `Sprintf`, `Errorf`, `Fprintf`. Format verbs: `%d` (int), `%s` (string), `%v` (default), `%+v` (with field names), `%#v` (Go syntax), `%T` (type), `%w` (wrap error).
* **`errors`** — `New`, `Is`, `As`, `Unwrap`, `Join` (combine multiple errors).
* **`strings`** — string manipulation.
* **`strconv`** — string ↔ number conversion.
* **`bytes`** — like `strings` but for `[]byte`.
* **`io`** — `Reader`, `Writer`, `Closer`, `Copy`, `ReadAll`. The Rosetta Stone of Go I/O.
* **`io/fs`** — virtual filesystem interface (since 1.16). Useful for embedding files into binaries.
* **`os`** — file and process operations: `Open`, `Create`, `Stat`, `Args`, `Getenv`, `Exit`.
* **`bufio`** — buffered I/O: `Scanner`, `Reader`, `Writer`. Use a Scanner for line-by-line reading.
* **`path`** and **`path/filepath`** — `path` for slash-separated paths (URLs); `filepath` for OS-aware paths.

### Concurrency and sync

* **`sync`** — `Mutex`, `RWMutex`, `WaitGroup`, `Once`, `Pool`, `Map`.
* **`sync/atomic`** — atomic operations on integers and pointers.
* **`context`** — `Background`, `WithTimeout`, `WithCancel`, `WithValue`.
* **`time`** — `Now`, `Sleep`, `Time`, `Duration`, `Ticker`, `Timer`. Time durations are typed: `5 * time.Second`, not `5000`.

### Network and HTTP

* **`net`** — TCP/UDP sockets, DNS resolution, IP addresses.
* **`net/http`** — HTTP client and server, both production-ready out of the box.
* **`net/url`** — URL parsing.
* **`net/http/httptest`** — test servers for testing HTTP handlers.
* **`encoding/json`** — JSON encoding and decoding.
* **`encoding/xml`**, **`encoding/csv`**, **`encoding/base64`**, etc. — other formats.
* **`encoding/binary`** — binary serialisation: ReadByte, ReadUint32 in little or big endian.

### Crypto

* **`crypto`** + many subpackages: `crypto/rand` (cryptographically secure RNG), `crypto/sha256`, `crypto/aes`, `crypto/tls`, `crypto/x509`. Don't roll your own crypto; use these.

### Database

* **`database/sql`** — SQL database client. Pluggable drivers (no driver in core; install separately).

### Reflection and runtime

* **`reflect`** — runtime type inspection. Used by JSON, ORMs, fancy generics. Slow; avoid in hot paths.
* **`runtime`** — runtime info: GOMAXPROCS, NumGoroutine, GC control. Mostly diagnostic.
* **`runtime/pprof`** — profiling.
* **`runtime/trace`** — execution tracing.

### Modern additions

* **`slices`** (1.21) — generic slice helpers.
* **`maps`** (1.21) — generic map helpers.
* **`cmp`** (1.21) — generic comparison.
* **`log/slog`** (1.21) — structured logging. Replaces older `log` for new code.

### What's *not* in the standard library

* No web framework (use `net/http` directly, or `chi` / `gin` for ergonomics).
* No ORM (use `database/sql` + a query builder, or `sqlx`, or `gorm` if you must).
* No advanced data structures (no `set`, no `priority queue` — write your own from a slice or a map).
* No async/await framework (use goroutines).
* No dependency injection framework (just construct things explicitly).

This minimalism is deliberate. The standard library has been called "boring," and that's a compliment.

## 3.8 Generics

Go added type parameters in 1.18 (March 2022). They're more limited than C++ templates or Rust generics, but they cover the common use cases.

```go
func Max[T cmp.Ordered](a, b T) T {
    if a > b {
        return a
    }
    return b
}

fmt.Println(Max(3, 5))           // 5
fmt.Println(Max("foo", "bar"))   // "foo"
fmt.Println(Max(3.14, 2.71))     // 3.14
```

The `[T cmp.Ordered]` declares a type parameter `T` constrained to types that satisfy `cmp.Ordered` — Go's built-in interface for "things that support `<`, `>`, `<=`, `>=`."

### Constraints

Constraints are interfaces. The `cmp.Ordered` interface looks like:

```go
type Ordered interface {
    ~int | ~int8 | ~int16 | ~int32 | ~int64 |
    ~uint | ~uint8 | ~uint16 | ~uint32 | ~uint64 |
    ~float32 | ~float64 | ~string
}
```

The `~int` syntax means "int and any type defined as `type Foo int`." Without `~`, only `int` itself would qualify.

You can also constrain to a method set:

```go
type Stringer interface {
    String() string
}

func PrintAll[T Stringer](items []T) {
    for _, item := range items {
        fmt.Println(item.String())
    }
}
```

This is just a regular interface used as a constraint. Any interface can be one.

The `any` keyword is the "no constraint" constraint:

```go
func First[T any](slice []T) (T, bool) {
    if len(slice) == 0 {
        var zero T
        return zero, false
    }
    return slice[0], true
}
```

`var zero T` gets you the zero value of `T` — useful when you need to return one without knowing the concrete type.

### Generic types

```go
type Stack[T any] struct {
    items []T
}

func (s *Stack[T]) Push(v T) {
    s.items = append(s.items, v)
}

func (s *Stack[T]) Pop() (T, bool) {
    if len(s.items) == 0 {
        var zero T
        return zero, false
    }
    n := len(s.items) - 1
    v := s.items[n]
    s.items = s.items[:n]
    return v, true
}

func main() {
    s := &Stack[int]{}
    s.Push(1)
    s.Push(2)
    v, _ := s.Pop()
    fmt.Println(v)   // 2
}
```

### When to use generics

* You really need to write the same code for many types and your only options are interface{} (loses type info) or copy-paste.
* The standard `slices`, `maps`, `cmp` packages are the clearest example: same code applies cleanly to many types.
* Custom data structures: `Stack`, `Queue`, `Tree`, `Set`. These genuinely benefit.

When *not* to use generics:

* When a regular interface works fine. Don't `func DoStuff[T MyInterface](x T)` when `func DoStuff(x MyInterface)` is simpler.
* When you only need it for one type. Just hard-code it.
* When the constraints get complicated. Go generics are deliberately less powerful than C++ templates; if you find yourself fighting them, the answer is usually "don't use generics here."

The Go community is still figuring out idioms for generics. Old code uses `interface{}` and reflection; new code is gradually adopting generics where appropriate. Don't go overboard.

## 3.9 What Production Go Looks Like

A few patterns and conventions you'll see in any serious Go project.

### Project layout

There's a community standard layout (see https://github.com/golang-standards/project-layout) but it's not mandatory. A common layout:

```
myservice/
├── cmd/
│   └── myservice/
│       └── main.go        # the binary entry point
├── internal/              # private packages
│   ├── server/
│   │   └── server.go
│   ├── orders/
│   │   └── orders.go
│   └── store/
│       └── store.go
├── pkg/                   # optional: public packages
│   └── client/
│       └── client.go
├── api/                   # protobuf, OpenAPI, etc.
├── deployments/           # k8s manifests, terraform, etc.
├── go.mod
├── go.sum
├── Makefile
└── README.md
```

`cmd/X/main.go` for each binary; everything else is library code under `internal/` (private) or `pkg/` (public). Many projects skip `pkg/` and put public stuff at the root.

### `Makefile`

A traditional convention:

```makefile
.PHONY: build test lint run

build:
	go build -o bin/myservice ./cmd/myservice

test:
	go test -race ./...

lint:
	golangci-lint run

run:
	go run ./cmd/myservice
```

### Dependency injection (the simple kind)

Go has no DI framework like Spring. Instead, construct things explicitly in `main`:

```go
func main() {
    cfg, err := config.Load()
    if err != nil {
        log.Fatalf("config: %v", err)
    }

    db, err := store.NewPostgres(cfg.DatabaseURL)
    if err != nil {
        log.Fatalf("db: %v", err)
    }
    defer db.Close()

    orders := orders.NewService(db)
    users := users.NewService(db)

    srv := server.New(cfg, orders, users)
    log.Fatal(srv.ListenAndServe())
}
```

Each component takes its dependencies as constructor arguments. You wire them up in `main`. For testing, you swap in mocks (which work because you're using interfaces). It's verbose but clear; a senior engineer can read `main` and understand the whole graph in a minute.

### HTTP server

The Go standard library's `net/http` is production-ready:

```go
package main

import (
    "encoding/json"
    "log"
    "net/http"
)

type Server struct {
    // dependencies...
}

type orderRequest struct {
    Price    int64 `json:"price"`
    Quantity int64 `json:"quantity"`
}

type orderResponse struct {
    ID int64 `json:"id"`
}

func (s *Server) handleCreateOrder(w http.ResponseWriter, r *http.Request) {
    var req orderRequest
    if err := json.NewDecoder(r.Body).Decode(&req); err != nil {
        http.Error(w, err.Error(), http.StatusBadRequest)
        return
    }

    // create the order...
    id := int64(42)

    w.Header().Set("Content-Type", "application/json")
    json.NewEncoder(w).Encode(orderResponse{ID: id})
}

func main() {
    s := &Server{}
    mux := http.NewServeMux()
    mux.HandleFunc("POST /orders", s.handleCreateOrder)

    log.Println("listening :8080")
    log.Fatal(http.ListenAndServe(":8080", mux))
}
```

The `"POST /orders"` syntax (Go 1.22+) is method-and-path routing built into the standard library. For more elaborate routing, middleware composition, etc., add `chi` or `gin`.

### Logging with `log/slog`

```go
import "log/slog"

slog.Info("request",
    "method", r.Method,
    "path", r.URL.Path,
    "user_id", userID,
)
```

`slog` produces structured logs (JSON or key=value text). Modern Go services should use it; older code uses `log` or third-party loggers like `zap` or `zerolog`.

### Configuration

Often via environment variables (12-factor pattern):

```go
import "os"

cfg.DatabaseURL = os.Getenv("DATABASE_URL")
cfg.Port = os.Getenv("PORT")
if cfg.Port == "" {
    cfg.Port = "8080"
}
```

For more elaborate config: `kelseyhightower/envconfig`, `spf13/viper`. But environment variables plus `os.Getenv` covers most cases.

## 3.10 Summary: The Rules

1. **Slices share backing arrays.** Modifying one can affect others. `append` may or may not reallocate.
2. **`copy` and `slices.Clone` for explicit copies.** Don't trust `append` for isolation.
3. **`make([]T, 0, cap)` to pre-allocate.** Avoids reallocation churn.
4. **Maps panic on write to nil.** Always `make` before writing.
5. **Map iteration order is random.** Sort keys for deterministic output.
6. **Maps aren't safe for concurrent writes.** Add `sync.RWMutex` or use `sync.Map` for the right access pattern.
7. **Strings are bytes, not characters.** Use `[]rune(s)` or `for i, r := range s` for code points.
8. **One concept per package.** Avoid `util`, `helpers`, `common`.
9. **Tests live next to code, file named `*_test.go`.** Use table-driven tests.
10. **Run tests with `-race`.** Run benchmarks with `-benchmem`.
11. **Use the standard library before reaching for dependencies.**
12. **Generics for genuinely generic code (data structures, slice/map helpers).** Don't over-apply.
13. **`internal/` is your friend.** Hide implementation details.
14. **Dependency injection by construction in `main`.** No frameworks needed.
15. **`net/http` and `log/slog` and `database/sql` are production-grade as-is.**

## Phase 1 Master Rules

A condensed reference, organised the way a senior Go engineer would think.

### The language

* Every type has a zero value; declared variables are immediately usable.
* No implicit numeric conversions. Be explicit.
* Functions can return multiple values. The error is conventionally last.
* No exceptions. Errors are values; check with `if err != nil`.
* No classes or inheritance. Use structs with methods, embedding, and interfaces.
* Methods have receivers. Pick value or pointer receiver consistently per type.
* Interfaces are implicit. A type satisfies an interface by having the right methods.
* Capitalised names are exported across packages. Lowercase are private.
* `go fmt`, `go vet`, `go test` are mandatory.

### Errors

* Return `(T, error)`. Wrap with `fmt.Errorf("...: %w", err)`.
* Sentinel errors as package-level vars: `var ErrFoo = errors.New("foo")`.
* Inspect with `errors.Is` and `errors.As`.
* Don't both log and return — pick one.
* Panic for genuine bugs; return error for runtime conditions.

### Concurrency

* Goroutines are cheap. `go func()` to spawn.
* Channels rendezvous unbuffered, queue when buffered. Default to unbuffered.
* Close from the sender side; only one closer; close once.
* `select` for multiple channel operations; with `default` for non-blocking.
* `for v := range ch` reads until the channel closes.
* "Don't communicate by sharing memory; share memory by communicating" — but mutexes are fine for state.
* Context for cancellation, deadlines, and request-scoped values.
* Always `defer cancel()` after `WithTimeout` / `WithCancel`.
* Maps are not safe for concurrent writes. Lock or use `sync.Map`.
* Run tests with `-race` in CI.

### Patterns

* Worker pool: N goroutines reading from one channel.
* Fan-in: many producers, merged via N goroutines into one channel.
* Pipeline: stages connected by channels, each stage owns one goroutine.
* Cancellation: `select` on `<-ctx.Done()` plus the work channel.
* Bounded concurrency: buffered channel as semaphore.

### Slices and maps

* Slices share backing arrays. Modify carefully.
* `append` may reallocate or not; don't rely on either.
* `slices.Clone` for safe duplication.
* Pre-allocate with `make([]T, 0, n)` when the size is known.
* Map iteration is randomised; sort keys for deterministic output.
* Always `make` a map before writing.

### Strings

* `len(s)` is bytes, not characters.
* `[]rune(s)` for code points.
* `for i, r := range s` decodes UTF-8.
* `strings.Builder` for repeated concatenation.

### Modules and packages

* `go mod init` to start, `go mod tidy` to clean up.
* One concept per package; package name is the directory's last element.
* `internal/` for private packages.
* Don't stutter: `http.Client` not `http.HTTPClient`.

### Testing

* Tests in `xxx_test.go`, alongside the code.
* `t.Errorf` for non-fatal, `t.Fatalf` for fatal.
* Table-driven tests with `t.Run` for subtests.
* `Example*` functions for verified documentation.
* `Benchmark*` with `b.N` and `-benchmem`.

### Standard library reflexes

* `fmt` for formatting, `strings`/`strconv` for string ops, `slices`/`maps` for collection helpers.
* `io.Reader`/`io.Writer` for streaming I/O. Compose them.
* `net/http` for HTTP. `database/sql` for SQL. `encoding/json` for JSON.
* `context` for cancellation. `sync` for shared state. `log/slog` for structured logging.

### Mindset

* Boring code is good code. Write the simplest thing that works.
* Don't fight the language. If a Java pattern doesn't fit, do the Go-native version.
* The standard library is comprehensive. Look there first.
* Concurrency is a tool. Reach for it when it solves a real problem; not before.
* Composition over abstraction. Inheritance was a mistake; Go doesn't make it.

### Success criteria

If after Phase 1 you can:

* Read an arbitrary Go file and understand its structure.
* Build a simple HTTP service with concurrent request handling and graceful shutdown.
* Choose between `chan`, `sync.Mutex`, `atomic`, and `sync.WaitGroup` for a given task.
* Spot a slice aliasing bug, a typed-nil-interface bug, or a goroutine leak in code review.
* Write idiomatic error handling with `%w` wrapping and `errors.Is` checking.
* Set up a Go module from scratch with tests, benchmarks, and CI.

Then you're a working Go programmer. From here, you specialise: web services (gin, chi, gRPC), distributed systems (etcd, raft, Kubernetes patterns), CLI tools (cobra, urfave/cli), data pipelines, observability tooling. Each is an afternoon to a week of focused study.

The language is small enough that there's no real "advanced Go." There's experienced Go, where you've seen the patterns enough times that you write idiomatic code without thinking. The path there is reps — read other people's Go, write your own, ship things, fix bugs.
